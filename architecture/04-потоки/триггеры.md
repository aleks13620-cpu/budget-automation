# Триггеры и потоки данных

## Пользовательские действия (триггеры из UI)

### Загрузка спецификации
**Триггер:** POST `/api/projects/:id/specifications` или `/bulk`
1. Multer сохраняет файл в `data/uploads/`
2. `sectionDetector` определяет раздел по имени файла и содержимому
3. `excelParser` / `gigachatSpecFromPdf` парсит содержимое
4. Позиции сохраняются в `specification_items`
5. Сохраняется снимок в `specification_items_history` (version=1, action='initial_parse')

### Загрузка счёта
**Триггер:** POST `/api/projects/:id/invoices` или `/bulk`
1. Multer сохраняет файл
2. `invoiceRouter` определяет стратегию: классика или GigaChat
3. Классика: `pdfParser` / `excelInvoiceParser` → если 0 позиций → GigaChat-fallback
4. GigaChat: `gigachatParser` (с кэшем по хэшу файла)
5. `invoiceValidator` оценивает качество → категория A/B/C
6. Позиции сохраняются в `invoice_items`
7. `supplier_parser_configs` проверяется — если есть сохранённый конфиг, применяется автоматически
8. Снимок в `invoice_items_history`

### Переразбор счёта (reparse)
**Триггер:** POST `/api/invoices/:id/reparse`
1. Читает `invoice_items_history` текущее состояние → сохраняет снимок «до»
2. Удаляет старые `invoice_items`
3. Применяет новый маппинг колонок к сохранённому файлу
4. Сохраняет новые позиции
5. Если конфиг помечен как «сохранить» — UPSERT в `supplier_parser_configs`

### Запуск сопоставления
**Триггер:** POST `/api/projects/:id/matching/run`
- `?mode=full` — удаляет все `matched_items`, пересчитывает с нуля
- `?mode=incremental` — сопоставляет только новые/несопоставленные позиции

Алгоритм (в `matcher.ts`):
1. **Tier 1 — артикул:** точное совпадение article → confidence 0.95
2. **Tier 2 — правила:** поиск в `matching_rules` по паре (spec_pattern, invoice_pattern, supplier_id) → confidence из правила
3. **Tier 3 — нечёткий поиск:** Dice-коэффициент по name ≥ 0.6 → confidence = score
4. **Tier 4 — name + chars:** combined similarity ≥ 0.5

Результат — список кандидатов в `matched_items`.

### Подтверждение совпадения
**Триггер:** PUT `/api/matching/:id/confirm`
1. Обновляет `matched_items.is_confirmed = 1`
2. Создаёт или обновляет запись в `matching_rules` (times_used++)
3. `constructionSynonymLearner` ищет аббревиатуры в паре → сохраняет в `construction_synonyms`

### Отклонение совпадения
**Триггер:** DELETE `/api/matching/:id`
1. Удаляет запись из `matched_items`
2. Если совпадение было из правила — создаёт negative-правило (`is_negative=1`)

### GigaChat-обогащение спецификации
**Триггер:** POST `/api/specifications/:id/gigachat-enrich`
- `?dryRun=true` — возвращает предпросмотр изменений, не сохраняет
- Без dryRun — применяет, сохраняет снимок, обновляет `spec_parse_rules`

### GigaChat-валидация совпадений (Tier 5)
**Триггер:** POST `/api/projects/:id/matching/validate-gigachat`
1. Берёт совпадения с confidence < порога (не подтверждённые)
2. Для каждой пары вызывает GigaChat (с кэшем в `gigachat_match_cache`)
3. Обновляет confidence для тех, где GigaChat подтверждает

### Применение скидки к счёту
**Триггер:** POST `/api/invoices/:id/apply-discount`
1. Сохраняет текущее состояние в `invoice_items_history`
2. Пересчитывает `price` для всех позиций: `price = price * discount_factor`
3. Устанавливает `invoices.discount_applied = discount_factor`

### Откат версии
**Триггер:** POST `/api/invoices/:id/rollback` или `/api/specifications/:id/rollback`
1. Читает нужный снимок из history
2. Удаляет текущие позиции
3. Восстанавливает из JSON-снимка

## Автоматические триггеры (без участия пользователя)

| Событие | Что происходит автоматически |
|---|---|
| Загрузка счёта от известного поставщика | Применяется `supplier_parser_configs` |
| Успешный парсинг GigaChat | Результат кэшируется в `gigachat_file_cache` |
| Подтверждение совпадения | Обновляется `matching_rules` + `construction_synonyms` |
| Удаление проекта | Каскадное удаление всех связанных данных (FK ON DELETE CASCADE) |
| Загрузка счёта | Автоматически создаётся supplier по имени файла (если не существует) |
