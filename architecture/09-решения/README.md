# Архитектурные решения

Здесь зафиксированы решения, видимые из кода — без домысливания причин.

## SQLite вместо PostgreSQL/MySQL

**Что сделано:** единственная база — SQLite-файл, синхронный драйвер better-sqlite3.
**Следствия:** простое развёртывание (нет отдельного сервера БД), нет параллельных записей из разных процессов (один инстанс), WAL обеспечивает конкурентные чтения.

## Нет ORM — прямой SQL

**Что сделано:** все запросы через `db.prepare(...).get/run/all`.
**Следствие:** полный контроль над SQL, нет абстракций, нет автомиграций.

## Fallback-цепочка парсинга

**Что сделано:** `invoiceRouter` пробует классику, при неудаче (0 позиций или нет финансовых полей) передаёт в GigaChat. Lite → Pro → Max при низком качестве.
**Следствие:** большинство счетов обрабатывается быстро и бесплатно; GigaChat тратится только на сложные случаи.

## Кэш GigaChat по хэшу файла

**Что сделано:** `gigachat_file_cache` хранит ответ по SHA256 содержимого файла.
**Следствие:** повторная загрузка одного файла не делает API-запрос; переразбор через UI использует кэш.

## Обучение через подтверждения (matching_rules)

**Что сделано:** каждое подтверждённое совпадение → правило в `matching_rules`; правило применяется в Tier 2 при следующих прогонах.
**Следствие:** система улучшается без переобучения модели; оператор не тратит время на одинаковые пары повторно.

## История версий через JSON-снимки

**Что сделано:** перед каждым изменением позиций (reparse, обогащение, скидка, rollback) текущее состояние сериализуется в JSON и сохраняется в `*_history`.
**Следствие:** простая реализация откатов без event-sourcing; но снимки могут занимать место при большом количестве итераций.

## State-based routing во фронтенде (без react-router)

**Что сделано:** текущая страница хранится в `useState` в `App.tsx`, URL не меняется.
**Следствие:** нет возможности сохранить/поделиться ссылкой на конкретный экран; упрощает реализацию, но ограничивает UX.

## Конфигурация парсера на уровне поставщика

**Что сделано:** после первой удачной конфигурации маппинга колонок оператор может сохранить её в `supplier_parser_configs`. При следующей загрузке счёта от того же поставщика конфиг применяется автоматически.
**Следствие:** ручная настройка нужна только один раз на поставщика.

## Файлы счетов не удаляются после парсинга

**Что сделано:** `data/uploads/` хранит оригинальные файлы бессрочно (удаляются только при удалении счёта/проекта).
**Следствие:** возможен повторный парсинг с другим маппингом; но диск расходуется.

## Express 5 (pre-release на момент разработки)

**Что сделано:** используется Express 5.2.1.
**Следствие:** `req.params.id` имеет тип `string | string[]`, код использует `String(req.params.id)` явно.
