## Часть 1. Внутренняя уборка (архитектура)

### 1.1. Единый центр парсинга

**Проблема:** Логика выбора парсера дублируется в двух местах:
- `invoiceRouter.ts` (строки 77–205) — fallback-цепочки для PDF, Excel, изображений
- `routes/invoices.ts` → `processInvoiceFile()` (строки 305–390) — своя fallback-цепочка

**Факт из кода:** `routeInvoiceFile()` вызывается ТОЛЬКО для картинок (`invoices.ts:180–182`). Для PDF и Excel он не используется — `processInvoiceFile` делает всё сам. Код invoiceRouter для PDF/Excel (строки 102–205) — **мёртвый код** в основном потоке.

При этом у `processInvoiceFile` логика богаче:
- fuzzy-поиск поставщика по БД (строка 341)
- проверка типа документа — не перезаписывать если GigaChat решил что это не счёт (строка 328)
- эвристики по поставщику (строка 301)
- Gemini OCR fallback для сканов (строка 381)

У `invoiceRouter.ts` ничего из этого нет.

**Решение: перенести полную цепочку из `processInvoiceFile` в `invoiceRouter.ts`.**

**Почему это лучший вариант (а не наоборот):**

| Критерий | invoiceRouter (сервис) | processInvoiceFile (route) |
|---|---|---|
| Место для бизнес-логики | ✅ Сервис — правильное место для решений "как парсить" | ❌ Route должен заниматься HTTP, валидацией, БД |
| Тестируемость | ✅ Можно вызвать отдельно, без HTTP-контекста | ❌ Нужно мокать request/response |
| Единый интерфейс | ✅ Уже возвращает `RouterParseResult` с source, category, confidence | ❌ Результат перемешан с DB-операциями |
| Переиспользование | ✅ Можно вызывать из benchmark-скрипта, из переразбора, из API | ❌ Привязан к конкретному HTTP-маршруту |

**Почему не "оставить как есть":** мёртвый код в invoiceRouter создаёт иллюзию, что он работает. Новый разработчик может начать править не тот файл — и правки не будут иметь эффекта.

**Почему не "после baseline":** baseline можно снять и сейчас — мы измеряем качество парсинга, а не рефакторим. Перенос логики делается ПОСЛЕ baseline, чтобы было с чем сравнивать.

**Порядок:**
1. Снять baseline (часть 2.2)
2. Перенести логику из `processInvoiceFile` (строки 305–390) в `invoiceRouter.ts`
3. `processInvoiceFile` вызывает `routeInvoiceFile()` и занимается только БД
4. Прогнать benchmark → убедиться, что числа не изменились

**Файлы:**
- `backend/src/services/invoiceRouter.ts` — расширить: добавить Gemini fallback, supplier fuzzy match, document_type guard
- `backend/src/routes/invoices.ts` — упростить: убрать строки 305–390, вызывать `routeInvoiceFile()`

### 1.2. OCR-цепочка (решение + обоснование)

**Текущий каскад в коде** (`invoices.ts:305–390`):
```
Нативный парсер (PDF text / Excel reader)
  ↓ если 0 позиций или нет финансовых данных
GigaChat (PDF или Excel)
  ↓ если PDF, GigaChat не помог, и нет текстового слоя
Gemini OCR (только для сканов)
```

**Решение: зафиксировать этот каскад как основной.**

**Почему не Gemini первым:**
- Gemini вызывается через OpenRouter → внешний API → задержка + стоимость
- На `Электротехмонтаж` Gemini даёт 0% по ценам (benchmark report) — значит, как основной парсер он ненадёжен
- GigaChat для структурированных PDF/Excel работает хорошо (есть GigaChat file cache для скорости)

**Почему не GigaChat первым (вместо нативного):**
- Нативный парсер бесплатный и мгновенный
- Для хорошо структурированных Excel-файлов (категория A) он даёт 100% результат без API-вызовов
- GigaChat нужен только когда нативный не справился

**Правило для Электротехмонтаж и подобных:**
- Gemini парсит текст/позиции (в этом он хорош)
- Цены берутся из GigaChat или ручного ввода (Gemini здесь ненадёжен)
- Это оформляется как отдельный режим: `gemini_text_gigachat_prices`

### 1.3. Место для ошибок (обоснование)

**Текущее состояние:**
- `FeedbackPage` — список ошибок по проекту, без группировки, без анализа
- `GlobalFeedbackPage` — то же самое, но по всем проектам. Просто список с кнопкой "Решено"
- `operator_feedback` в БД хранит: тип, проект, позицию спецификации, позицию счёта, комментарий, статус
- **Нет поля `supplier_id`** в operator_feedback → нельзя даже группировать по поставщику без JOIN

**Почему экран анализа нужен (а не SQL-запросы):**

| Критерий | Экран в UI | SQL-запросы вручную |
|---|---|---|
| Кто пользуется | Оператор (Иван, Сергей) | Только разработчик |
| Скорость | Открыл → увидел паттерн | Написал запрос → разобрал результат |
| Регулярность | Смотрят после каждого проекта | Когда "накипело" |
| Комфорт сотрудникам | ✅ | ❌ |

**Решение: расширить существующий `GlobalFeedbackPage`, а не делать новый экран.**

Почему: инфраструктура уже есть (API `/feedback/all`, компонент, фильтры). Нужно добавить:
- JOIN на suppliers через invoice_items → invoices → suppliers
- Группировку: "Поставщик → тип ошибки → количество"
- Простую визуализацию: "Электротехмонтаж: 8 ошибок цены, 2 ошибки количества"

**Файлы:**
- `backend/src/routes/matching.ts` — добавить endpoint `/feedback/analysis`
- `frontend/src/pages/GlobalFeedbackPage.tsx` — добавить вкладку "Анализ"

## Часть 2. Процесс обучения

### 2.1. Эталон (уже есть)

12 JSON в `scripts/benchmark-ready/` — утверждённые "правильные ответы".
Содержат: поставщик, позиции (название, артикул, цена с НДС, количество).

**Ожидание:** Иван и Сергей подтверждают корректность этих 12 JSON.

### 2.2. Цикл обучения парсинга (OCR-качество)

```
 ┌──────────────────────────────────────────────────────────────┐
 │  1. Прогнать 12 эталонных счетов через систему               │
 │  2. Benchmark-скрипт сравнивает результат с эталоном          │
 │  3. Отчёт: по каждому поставщику —                            │
 │     % совпадения названий / цен / количества                  │
 │  4. Найти провалы (пример: «Электротехмонтаж, цены 0%»)      │
 │  5. Разобрать причину провала:                                │
 │     — парсер не увидел текст? → проблема OCR                  │
 │     — цена прочитана без НДС? → проблема VAT-детектора        │
 │     — строки обрезаны? → проблема chunking                    │
 │  6. Внести точечную правку                                    │
 │  7. Прогнать benchmark заново → проверить:                    │
 │     — провал исправлен?                                       │
 │     — другие поставщики не упали больше чем на 2%?            │
 │  8. Записать правку и новый отчёт в docs/benchmark-reports/   │
 └──────────────────────────────────────────────────────────────┘
```

**Пороги:** названия ≥ 90%, цены с НДС ≥ 95%, единицы ≥ 90%.

**Инструмент:** доработать существующую Python-цепочку (`process_audit_files.py` → `02_benchmark_gemini.py` → `03_score_results.py`), потому что она уже умеет сравнивать и считать метрики. Писать заново на TypeScript нет смысла — пороги и формулы уже реализованы.

### 2.3. Цикл обучения матчинга (сопоставление позиций)

В системе уже работают три механизма обучения:

**Механизм 1: matching_rules** (`matcher.ts:355–411`)
- Оператор подтверждает матч → пара "паттерн спецификации ↔ паттерн счёта" сохраняется с confidence 0.92
- При следующем матчинге система пробует эти правила первыми (до fuzzy-поиска)
- Привязаны к поставщику (`supplier_id`) → учатся по каждому поставщику отдельно

**Механизм 2: construction_synonyms** (`constructionSynonymLearner.ts`)
- Из каждого подтверждённого матча автоматически извлекаются пары "аббревиатура ↔ полная форма"
- Пример: "кш" → "кран шаровой"
- Используются при нормализации текста перед матчингом

**Механизм 3: spec_parse_rules** (`gigachatSpecParser.ts:197–207`)
- Оператор поправляет обогащение спецификации → пара "было → стало" сохраняется
- При следующем обогащении GigaChat получает эти правила в промпт
- Взвешены по `times_used` — чем чаще правили, тем выше приоритет

**Полный цикл для оператора:**
```
 ┌──────────────────────────────────────────────────────────────┐
 │  1. Загрузить счёт + спецификацию в проект                   │
 │  2. Система автоматически сопоставляет позиции                │
 │  3. Оператор проверяет:                                      │
 │     ✅ Подтвердил правильный матч                             │
 │        → matching_rules запомнила паттерн                     │
 │        → synonyms извлекли аббревиатуры                       │
 │     ❌ Отклонил неправильный матч                              │
 │        → negative rule: эту пару больше не предлагать         │
 │     ✋ Создал матч вручную                                     │
 │        → новое правило с confidence 0.92                      │
 │  4. Следующий счёт от ТОГО ЖЕ поставщика:                    │
 │     система сначала пробует накопленные правила               │
 │  5. С каждым проектом % автоматического сопоставления растёт  │
 └──────────────────────────────────────────────────────────────┘
```

**Чего НЕ хватает сейчас и что нужно добавить:**

| Что | Статус | Что делать |
|---|---|---|
| matching_rules работают | ✅ В коде | Проверить benchmark'ом |
| construction_synonyms извлекаются | ✅ В коде | Проверить: не создают ли ложные пары? |
| spec_parse_rules подаются в GigaChat | ✅ В коде | Проверить: улучшают ли обогащение? |
| Оператору показывается "система запомнила" | ⚠️ Частично | Есть метка learned_rule, нет полного UX |
| Benchmark для матчинга (эталонные пары) | ❌ Нет | Сделать после стабилизации парсинга |
| Оператор видит, ПОЧЕМУ система предложила матч | ❌ Нет | Добавить: "совпал по правилу / по названию / по артикулу" |

### 2.4. Что после 12 эталонных файлов

**Решение: ручное добавление по решению владельца** (кнопка "Экспорт в benchmark").

**Почему не автоматически:**
- Оператор может подтвердить матчи "на автомате", не проверив тщательно — плохие данные загрязняют эталон
- Эталон должен быть ЛУЧШЕ системы — иначе benchmark теряет смысл
- Владелец знает, какой проект был разобран тщательно, а какой — "по-быстрому"

**Почему не "только 12 навсегда":**
- 12 счетов от ~10 поставщиков — узкая выборка. Система может идеально работать на них и проваливаться на новых.
- Цель: 30–50 эталонов от 15–20 поставщиков

**Как это работает:**
1. Владелец выбирает проект, где все матчи проверены тщательно
2. Нажимает "Экспорт в benchmark" → система генерирует JSON в формате `benchmark-ready/`
3. JSON добавляется в `scripts/benchmark-ready/`
4. При следующем прогоне benchmark включает новый эталон

**Файлы:**
- `backend/src/routes/export.ts` — добавить endpoint экспорта в benchmark-формат
- `frontend` — кнопка на странице проекта (видна только владельцу/админу)

### 2.5. Обратная связь по обучению

**Что видит оператор сейчас:** метка `learned_rule` у матча — и всё.

**Что должен видеть:**
- При матчинге: "Сопоставлено по правилу (поставщик X, использовано 5 раз)" / "По названию (сходство 87%)" / "По артикулу (точное)"
- При обогащении: "Учтено 12 ваших прошлых исправлений для этой спецификации"
- После подтверждения матча: "Правило сохранено. В следующий раз система сопоставит эту пару автоматически"

Это не новая функциональность — это отображение данных, которые система уже считает (`match_type`, `confidence`, `times_used`).
