## 1. СТРАТЕГИЧЕСКИЙ ФОКУС — обучение по метрикам (новый эпик)

**Концепция (со слов владельца):** двигатель проекта — обучение системы через оператора. Чтобы им управлять, метрики обучения должны быть **постоянно на виду** и улучшаться с каждым действием оператора. Нужен **дашборд динамики обучения**.

**Петля:** оператор подтверждает/отклоняет/связывает спорные позиции → система учит правила+синонимы → авто-сопоставление растёт на новых данных. Доказано: проект 6 вырос 77% → 88.3% после ~300 операторских действий (`reflection_2026-05-27_training_works_in_sample`).

### Какие метрики выводить
| Метрика | Зачем | Источник сейчас |
|---|---|---|
| Покрытие = matched/total, по проекту | Главный KPI сопоставления | ✅ `/api/projects/:id/matching/stats` |
| Подтверждено (confirmed) | Сколько обучающего сигнала дал оператор | ✅ тот же эндпоинт |
| # выученных синонимов / правил (глобально) | Рост «памяти» системы | ⚠️ есть в БД (`construction_synonyms`, `matching_rules`), нет эндпоинта |
| **Динамика во времени** (снимки) | Видеть, что метрики РАСТУТ | ❌ нет истории/снапшотов |
| **Прирост на действие оператора** | «Каждое действие что-то меняет» | ❌ нет |
| **Перенос обучения** (учим проект A → растёт авто-матч на B до правок оператора) | Доказать, что обучение генерализуется | ❌ нет |

### Что уже есть (investigation-first — НЕ строить заново)
- Live-точки: `/api/projects/:id/matching/stats`, `/api/projects/:id/summary`.
- Dev-отчёт: `scripts/feedback-report.py`.
- **Фаза 0 выполнена:** ветки `step-2a-quality-monitor`/`step-7b` пусты относительно main; существующий `scripts/quality-monitor.py` — это **парс-качество OCR** (имена/цены vs эталон), НЕ метрики обучения. Инструмента «динамика покрытия по действиям оператора» НЕТ → строим.
- **Ключ для дашборда уже в API:** `/api/projects/:id/matching` → `summary.tierBreakdown` (llm_suggestion / name_similarity / learned_rule) + `total/matched/confirmed`. Не хватает только **истории во времени** и **страницы-графика**.

### Что не хватает (gap)
1. **Персистентность** — таблица снапшотов метрик (на каждый матч/действие или по cron).
2. **Видимость** — страница-дашборд (графики покрытия/confirmed/синонимов во времени).
3. **Перенос** — метрика «авто-матч на проекте до операторских правок» (raw baseline) против «после переноса правил с других проектов».

### Предлагаемый порядок (investigation-first)
- **Фаза 0 (исследование):** ревизия `step-2a-quality-monitor` / `step-7b-field-quality-report`; решить — оживляем или строим минимально.
- **Фаза 1 (снапшоты):** лёгкая запись метрик в БД при каждом матче/действии (без изменения matcher.ts).
- **Фаза 2 (дашборд):** страница с графиком динамики.
- Гейт каждой фазы — sub-agent + pre-deploy-check. Регрессия проекта 6 (333/268/44) не падает.

**Статус (обновлено 2026-06-03):** **Фаза 1 «Захват» ЗАДЕПЛОЕНА** (`6f02bc9`) — таблица `metric_snapshots` + снимки на прогон / действие оператора / старт / ежедневно. pre-deploy 5-move сошёлся за **3 круга** (loop-until-clean поймал 4 snapshot-bypass пути: unconfirm, select-override, manual-match reuse, gigachat-validate). Build+smoke зелёные, регрессия 6/11 не дрогнула. Дальше — **Фаза 2** (эндпоинт чтения истории) → **Фаза 3** (страница-график).

**CARRY (метрики, не блокеры):**
- **C1:** bulk-циклы (bulk-confirm/reject/analog, group-followers) пишут N снимков на одно bulk-действие → снимок один раз после цикла (perf + меньше дублей в таймлайне).
- **C2:** нет прунинга `metric_snapshots` → ретеншн/чистка при росте.
- **C3:** memory-счётчики (`source='learned'`) в одном try с coverage-счётчиками — при гипотетическом отсутствии колонки уронят весь снимок; изолировать.

## 2. «ЛАСТОЧКА ОВ» (проект 11) — разбор матчинга → РЕГРЕССИИ НЕТ

**Жалоба:** до правок было ~68%, при повторном запуске — 29% из 328 позиций.

**Факты (перепроверено с прода 2026-06-03, read-only):**
- **РЕГРЕССИИ НЕТ. Текущее покрытие — 211/328 = 64.3%** (`matching/stats` + `summary.tierBreakdown`), статус `idle` (прогон завершён). ≈ запомненным «68%».
- Напугавшее «**29%**» (98/328) — снимок на **СЕРЕДИНЕ прогона**: быстрые матчи по похожести посчитались, а медленный **LLM-проход (GigaChat) ещё не дописал 200+ подсказок**. Дописал → 64%. Урок `reflection_2026-05-17`: проверять завершённость прогона до выводов. **(Ранняя гипотеза «сброс обучения через re-upload» — НЕВЕРНА, снята данными.)**
- **Состав матчей:** llm_suggestion **207**, name_similarity **4**, **learned_rule 0**. → Правила/синонимы с проекта 6 **НЕ переносятся** на ИТЕСА/АС ГРУПП/TEPLA (другой товарный домен). 64% держит **LLM-сэмплер** (он обобщает на новые домены); learned-правила домен-специфичны и появятся ПОСЛЕ операторских подтверждений на этих поставщиках.
- **Дубли (#15):** 234 из 328 позиций — в **62** группах → решений оператора ≈ **156** (94 одиночных + 62 лидера).
- **Несопоставлено 117:** значимая часть — заголовки («1 этаж») и под-варианты радиаторов («Левое исполнение. Нижнее подключение dп=20 мм»), которые **структурно не матчатся отдельно** (описательные/дочерние строки), а не «нужно правило». Потолок покрытия < 100%.
- Данные чистые (ИТЕСА без глифов), парс корректный — **CURRENT_BUG ≈ 0**. Базовый снимок: `references/2026-06-03_lastochka_baseline_snapshot.md`.

**Вывод:** **проблемы-регрессии нет.** Ласточка — непрогретый проект на **64% (весь матч от LLM, 0 подтверждений)**. `confirmed=0` = оператор её ещё не подтверждал (либо подтверждения и не делались, либо стёрлись при re-upload — отличить нельзя, на путь вперёд не влияет).

**Decision points (нужно решение владельца):**
- **A. Восстановить потерянные подтверждения** из прод-бэкапа (`/root/.../*.bak-20260527-pre-rematch`, `*.bak-20260528-pre-cleanup`)? Риск: бэкапы от 27–28 мая, могут предшествовать работе по Ласточке/фиксу ИТЕСА. Сложно, может не помочь.
- **B. (рекомендую) Переобучить начисто** через оператора на уже чистых данных → покрытие вырастет, и **использовать Ласточку как живую витрину дашборда метрик** из эпика №1.
- Параллельно: прогнать **#15 дубль-группировку** на Ласточке; проверить **перенос синонимов** с проекта 6 (применяются ли к ИТЕСА/АС ГРУПП).

**Зачем нужен дашборд (подтверждено этим эпизодом):** точечный замер «29%» ввёл в заблуждение — финал был 64%. Дашборд показывает завершённость прогона и динамику → «ложной тревоги» и «тихой» потери обучения больше не будет.

**Статус:** разбор СДЕЛАН (регрессии нет). Рекомендация — **B: закрепить 64% через оператора** (подтверждения → learned-правила для ИТЕСА/АС ГРУПП), снимая метрики по ходу.
