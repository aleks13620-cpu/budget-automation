## Что сделано (по скоупу брифа)

### 1. Расширено распознавание токенов признаков
- **Кириллический «Дн»/«дп» → DN.** Новое правило в `normalizeEngineeringTokens` (требует цифры после, чтобы «дно»/«переходной» не трогать).
- **`extractDnValue` стал DB-free** (через новый `normalizeStructuralTokens` = `removeGostBrackets` + `normalizeEngineeringTokens`) и теперь терпит:
  - суффикс размера, приклеенный к цифрам: `dn57x3.5` → 57 (раньше `\b` после цифр ломался об `x` → возвращал `null` даже для латиницы);
  - сепараторы: `DN-50`, `DN. 50` → 50 (как раньше делал полный пайплайн через strip пунктуации).
- **Новая чистая функция `extractMarkingFeatures(text)`** (экспортирована, DB-free, юнит-тестируема) извлекает:
  - `dn` — диаметр (DN/Ду/Дн/дп/ø);
  - `cross` — сечение «AxB», напр. `57x3.5`;
  - `config` — многоместный конфиг «NN-NN-NN» (тройник переходной 16-20-16 vs равнопроходный 16-16-16);
  - `marks` — короткие латинские буквенные маркировки (CV/CVL/C); бренды Title-case и коды, приклеенные к цифрам (DN15/PN16), не ловятся;
  - `sizes` — «голые» целочисленные размеры (фиксаторы 16/20/25), исключая числа, уже учтённые в dn/cross/config; **номера ГОСТ/ТУ/DIN вырезаются** (`STANDARD_REF_RE`), чтобы не считались размерами.

### 2. Штраф за рассогласование — зеркалит и обобщает DN-штраф
`getDnScore` → переименована в **`getStructuralScore`** и обобщена на все измерения. Логика:
- **CONFLICT** (оба несут признак, но РАЗНЫЕ) → `-1` (доминирует — «не тот вариант»);
- **AGREEMENT** (хоть одно измерение совпало, конфликтов нет) → `+1`;
- иначе односторонний пропуск DN (спека DN-типизирована, в счёте DN нет): `-1` **только если** счёт вообще не упоминает это число (иначе голый размер = тот же диаметр → `0`) — сохраняет ловлю «труба vs теплоизоляция»;
- иначе `0`.

**Штраф не дублировался отдельной строкой — он влит в существующий `dnScore`.** Поле `dnScore` — это и есть структурный дискриминатор; имя сохранено для совместимости с потребителями ранжирования. Поэтому ВСЕ существующие места уже работают без правок:
- `matcher.ts` штраф `dnScore===1 → +0.1`, `dnScore===-1 → −0.18` (matcher.ts:~592-593) — «жёсткий штраф»;
- `matcher.ts` первичный ключ сортировки кандидатов (`b.dnScore - a.dnScore`);
- `routes/matching.ts` выбор авто-#1 (`bestCandidateBySpec`, :487-492);
- dev-тулзы `replay-matching.ts`, `bench-matching.mjs`.

**Почему влил в `dnScore`, а не добавил параллельный `markingScore`:** `dnScore` — ПЕРВИЧНЫЙ ключ ранжирования и в matcher-сортировке, и в выборе авто-#1 в роуте. Отдельный score, не добавленный в этот ключ, позволил бы кандидату с верным DN, но НЕВЕРНОЙ маркировкой обойти верный кандидат по первичному ключу. Влитие гарантирует демоушн «не того варианта» во ВСЕХ путях ранжирования. Бонус: **миграция БД не нужна** — `dnScore`/`quantityScore` не персистятся (в `matched_items` пишется `confidence`, не они).
