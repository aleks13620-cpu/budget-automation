## 0. Что уже есть (фундамент, не строить заново)
- **Захват замечаний** — теги (`POST /projects/:id/feedback/tag`, safelist `ALLOWED_TAGS` + тег `question`) и свободный текст (`POST /projects/:id/feedback`, `type='error_report'`). Пишется в `operator_feedback` со `status='new'`.
- **Вывод** — `GET /feedback/all` (заметки + теги, со статусом), `GET /feedback/export` (xlsx), `PATCH /feedback/:id/resolve` (ставит `resolved` одной записи).

- **Learner НЕ читает `operator_feedback`** — теги/замечания это сигнал «посмотрите», не обучающее правило.

## 0.1 Проблема (почему сейчас не работает как цикл)
Сигнал **фиксируется и лежит**. Чтобы его увидеть — надо ЗАЙТИ и открыть страницу (pull). Нет: (а) пуша «появилось новое», (б) механизма, чтобы агент-разборщик **сам** видел очередь, (в) ясных ролей «кто чинит», (г) сводной динамики «насколько близко к финишу». Итог — оператор звонит, контекст теряется, автоматизации нет.

| **Владелец** | ты | направление, приоритеты, **«ок» на изменения**, приёмка | решения |

| **Уведомитель** (служебный) | бэкенд-хук + Telegram-бот | при новом замечании шлёт пуш (минимальный контент) | только исходящее уведомление |

**Имя агента-разборщика — «Арта» (ИИ)** [решение владельца 2026-06-03]. Честно про неё: Арта не сервис, живущий на проде. «Мониторит постоянно» = одно из двух механизмов: **(C1) при заходе в проект** ей в контекст подтягивается очередь (хук), и/или **(C2) по расписанию** (cron-routine) она сама проверяет прод. Поведение настраивается хуком + инструкцией-памяткой агенту «Арта».

> Действие: после подтверждения ролей — заменить слово «команда» в `operator_feedback_memo.md` на конкретные роли.

## 2. Целевой автоцикл замечаний

```
Оператор кликает замечание
        │
        ▼
[A] Запись operator_feedback (status=new)         ← уже есть
        │
        ├──► [B] Уведомитель → Telegram (мин. контент: проект, № позиции, ярлык, ссылка)   ← НЕ построено
        │
        ▼
[C] Очередь неразобранного (/feedback/all?status=new)   ← данные есть
        │       ├─ C1 SessionStart-хук: агент при заходе ВИДИТ очередь (в контексте)
        │       ├─ C2 cron-routine: агент сам проверяет раз в N ч → пуш/разбор
        │       └─ C3 (опц.) живой файл docs/.../open_feedback.md
        ▼
[D] Триаж: классификация замечания
        ▼
[E] Фикс (через dev+гейт) ИЛИ эскалация в план (PRB)   +   [F] статус → resolved
```

### [B] Уведомление (из брифа Ч3 — переиспользуем)
- Канал: **Telegram-бот** в рабочую группу (достижимость с РФ-сервера проверена 01.06). Fallback — РФ-SMTP / только in-app бейдж.
- **Контент-политика (residency, критично):** наружу — ТОЛЬКО проект + № позиции + ярлык тега + ссылка в инбокс. **БЕЗ названий материалов/поставщиков** (правило «данные на РФ-серверах»). Полный контент — в системе.
- Каденс [решение владельца]: **сразу на каждое замечание**, сообщение несёт **статус**; при закрытии — **ответное «✓ отработано»** (двусторонний цикл: оператор/группа видят, что сигнал реально обработан, а не просто «улетел»). Токен/chat_id — в `.env`, не логировать.

### [C] Видимость для агента (НОВОЕ — ядро твоего запроса «агент видит неразобранное»)
- **C1 — SessionStart-хук** (`settings.json`): при старте сессии в этом проекте хук дёргает `/feedback/all?status=new` и вкидывает мне в контекст список неразобранных. → «при заходе агент сразу видит, что разобрать». Низкий риск, чисто read-only.
- **C2 — cron-routine** (через механизм запланированных агентов): раз в N часов я сам проверяю прод; есть новые → пуш и/или открываю разбор. Делает мониторинг проактивным, без участия человека.
- **C3 — живой файл** `docs/plans/active/open_feedback.md`: всегда свежий список (генерится хуком/cron). Удобно как «вики прокинут».

### [D] Триаж — классификация (определяет, кто и как чинит)

| Класс | Пример тега | Кто решает | Авто-фикс? |
|---|---|---|---|
| Данные / парсер | «парсер пропустил», битые имена/глифы | агент-разборщик | предложить фикс → **гейт → твой «ок»** (forward-only) |
| Матчинг (не то сопоставление) | «не та маркировка», «нужны альтернативы» | оператор (отклонить/аналог); агент учит правило | оператор решает; правило — через гейт |
| Цена / бизнес-факт | «цена не та», «не покупали» | человек (проверить источник) | **не авто** |
| Вопрос оператора | «вопрос / не уверен» | человек (владелец/эксперт) | **не авто** — нужен ответ |
| UX / баг системы | «кнопка путает», как FB-STATUS | агент-разборщик | фикс → гейт |

**Граница автономии Арты [решение владельца 2026-06-03]:**
- **Безопасные классы** (данные/парсер, UX-баги), где фикс **проверен 5-move гейтом до PASS** → Арта чинит **и деплоит сама**, без отдельного «ок».
- При **малейшей развилке** в принятии решения (неоднозначность, выбор из вариантов, риск побочки, неуверенность в классе) → **СТОП, запрос владельцу.**
- **Рискованное** (что течёт в обучение/данные — `feedback_no_corrupt_through`) → **никогда не авто**, всегда через владельца + ревью.
- Инвариант: 5-move гейт PASS обязателен в любом случае; «авто» = Арта прогоняет гейт сама и при чистом PASS + отсутствии развилки идёт дальше.

### [F] Статус и баг FB-STATUS
- **Баг:** `GET /projects/:id/feedback` не возвращает `status` → на странице проекта «разобрано» не отображается/не переживает перезагрузку. Чинится одним полем в SELECT.
- **UX:** кнопка «✓ Разобрано» и лейбл статуса «✓ Разобрано» выглядят одинаково → путает. Переименовать кнопку в «Отметить разобранным».
- **Эскалация:** статус «→ задача» (ссылка на реестр проблем PRB), чтобы повторяющееся уходило в план, а не терялось.

## 3. Динамическая финиш-метрика (по всем проектам)

**Цель оператора** — готовая спецификация. Нужна цифра «насколько близко к финишу» — **не галочка, а живой тренд** (оператор может не дойти, ошибиться, переделать, перезалить — линия и падает, и растёт).

- **Хорошая новость:** динамика уже копится в `metric_snapshots` (total/matched/confirmed по времени на каждое действие). Финиш-метрика — это в основном **новый ВЗГЛЯД (агрегация) на уже собранные данные**, а не новый сбор.
- **Формула (черновик, на подтверждение):**
  `готовность% = (подтверждено + явно-разобрано) / (всего ЗНАЧИМЫХ позиций)`,
  где «значимые» = total − структурно-несопоставимые (заголовки «1 этаж», под-строки). v1 можно без вычета (значимые=total), приняв что 100% недостижимо, и смотреть **тренд**, а не абсолют.
- **Сводно по всем проектам:** глобальная готовность = Σподтверждено / Σзначимых + **график во времени** (живая история). Плюс таблица по проектам «сколько осталось до финиша».
- **Подача:** расширить существующий дашборд блоком «Все проекты» (агрегат + тренд готовности).

## 5. Ограничения и риски (честно)
- **SEC-1:** прод `/api` без токена (открыт). Уведомления/cron-поллинг и хук работают и так, но это общая дыра — возможно, F3/F5 — повод закрыть API (отдельное решение).
- **Residency:** наружу (TG) — только ID+ссылка, без бизнес-текста. Жёстко в Move 5 гейта.
- **`feedback_no_corrupt_through`:** агент не вливает в обучение/данные рискованное без ревью.
- **Что я есть:** «постоянный мониторинг» = хук при заходе + cron, не живой демон.
- **Привычка > код:** цикл поедет, только если оператор кликает, а не звонит (память: `reflection_2026-05-27_feedback_ux_gap`).
