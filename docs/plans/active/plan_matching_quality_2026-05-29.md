## План — порядок выполнения

### 🟡 Пакет B — Signal-first (СЕГОДНЯ)
**Задачи:** #14 (top-N альтернатив) + #17 (quick-tags)
**Цель:** структурированный сигнал от оператора + видимость альтернатив

**Acceptance criteria:**
- #14: На странице «Сопоставление» для каждой спеки видны top-3 кандидатов по confidence в свёрнутом виде; на «три точки → ручная связка» по-прежнему работает.
- #17: Inline-теги «цена не та / не та маркировка / нужны альтернативы / дубль / не покупали / аналог / парсер пропустил» на каждой строке матча. 1 клик → запись в `operator_feedback` с типом `tag_<name>`. Старая форма «Замечание» остаётся для произвольного текста.

**Метрика успеха:**
- Деплой за день
- Sub-agent верификация PASS после каждой из 4 фаз
- Регрессия 333/268/44 не падает после рематча

**Процедура:** 4 фазы (Design → Implement → Deploy → Post-deploy verify), sub-agent после каждой.

### 🟢 Пакет A — Productivity (СЛЕД. ДЕНЬ ИЛИ ПОСЛЕ B)
**Задачи:** #15 (детектор дублей) + (#14 уже сделан в B)
**Цель:** оператор не подтверждает одно и то же 5-7 раз

**Acceptance criteria:**
- #15: При запуске матчинга алгоритм группирует спеки по нормализованному имени; в UI оператор видит «лидера» с подсказкой «×N копий», подтверждает один раз, остальные `is_confirmed` ставятся автоматически с тем же матчем. Учитель синонимов вызывается **один раз** на группу, а не N раз.
- Количество в спецификации сохраняется (суммируется или показывается по системам).

**Метрика успеха:**
- На повторном открытии проекта 6 виден свёрнутый блок «137 дубль-строк объединены в 90 групп»
- Sub-agent верификация PASS, тиры в API не меняются

## Процедура и правила

Для каждой задачи применяем:

### 4-фазный цикл с sub-agent верификацией
1. **Phase 1 (Design)** — read existing code, design solution, sub-agent проверяет дизайн
2. **Phase 2 (Implement)** — edit код, build, smoke test, sub-agent проверяет diff+smoke
3. **Phase 3 (Deploy)** — regression check, pre-deploy-check skill, commit+push, sub-agent проверяет hygiene
4. **Phase 4 (Post-deploy)** — verify deploy, regression на проде, MEMORY обновление, sub-agent проверяет финал

### Правила проекта (обязательные)
- `feedback_no_action_without_confirmation` — никаких действий без явного «ок»
- `feedback_explain_before_after` — состояние → изменение → результат до действия
- `feedback_build_before_push` — `npm run build` чистый перед push
- `feedback_commit_named_files` — git add только именованные файлы
- `pre-deploy-check skill` — обязательный гейт перед `git push origin main`
- `feedback_no_hardcode` — никаких хардкодов

### Контрольные точки для пользователя

После каждой фазы — короткий рапорт sub-agent (PASS/FAIL + 1-2 строки) + ожидание «ок» перед следующей фазой.

## Decision points в этом плане
