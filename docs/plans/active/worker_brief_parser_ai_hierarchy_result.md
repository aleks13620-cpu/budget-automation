## Почему литеральная цель «>80% всех строк» недостижима (честно)
Бриф-цифры «~3% → >80%» были откалиброваны против ПРОД-пайплайна на pdfplumber (737 «грязных»
строк, привязка только по номеру → parent_item_id 10/328 ≈ 3%). Изолированный замер выше идёт по
ЧИСТОЙ экстракции Gemini (147 строк), где:
- ~20% строк (30/147) — законные **нумерованные родители** разных типов радиаторов (C11/C21/…), они по определению не «привязаны» (они корни);
- максимум «привязанных среди всех строк» ≈ 117/147 = **79.6%** даже при идеальной привязке — то есть >80% арифметически невозможно при такой доле родителей;
- поэтому корректная метрика — **доля привязанных ДЕТЕЙ** (76%→87%) и **число вредных сирот-фрагментов**, а не «% всех строк».

## Гейт #4 — фикс ложной блокировки (важно)
Первая версия гейта блокировала по `orphanFraction > 0.5` и **ложно блокировала Сокольи ВК**
(плоский список, 100% сирот, но имена самодостаточны). Исправлено: гейт теперь блокирует по
**`bareOrphanFraction > 0.5`** — доле «голых» сирот (бессмысленные без родителя коды/типоразмеры:
«C11-300-500», «DN15», «То же»; длинные описательные имена НЕ считаются «голыми»). Проверено
детерминированно (в харнессе, секция «Гейт hardBlock»):
- катастрофа (DN-коды без родителя): bareOrphan 100% → **hardBlock=true** (блок) ✓
- плоский список длинных имён: bareOrphan 0% → hardBlock=false ✓
- реальный Сокольи: bareOrphan 0% → **hardBlock=false (не блокируется)** ✓
- реальный Ласточка (хорошая): bareOrphan 2.7% → hardBlock=false ✓

## Риски и заметки для оркестратора
1. **Порог гейта `bareOrphanFraction>0.5` — это «катастрофа = большинство строк голые коды».** Частичный развал (напр. 24% голых сирот) НЕ блокируется, остаётся advisory `suggestElevatedReview`. Если владелец хочет более раннюю жёсткую отсечку — снизить порог. Сейчас сознательно консервативно (лучше пропустить с review-флагом, чем ложно заблокировать валидную спеку).
2. **GigaChat вызывался вживую и вернул 402 (Pro/Max) + таймаут (базовая)** → отработал Gemini-фолбэк. На проде это текущий реальный путь. Поведение нового якоря от провайдера LLM не зависит (он читает JSON-поля `parent_position`/`parent_name_hint`). Если на проде восстановят тир GigaChat — якорь так же сработает на его ответе.
3. **Стоимость:** Gemini-фолбэк шлёт PDF (base64) на OpenRouter. Включается только когда pdfplumber дал низкое качество (`noPosFraction>0.5`) ИЛИ GigaChat недоступен/ошибся — на «хороших» спеках с номерами LLM не зовётся.
4. **Кэш парсера v4→v5** — старые закэшированные результаты спек на проде станут невалидны (одноразовый ре-парс при следующей загрузке/реуплоаде).
5. **Сокольи остаётся плоским (linked≈0).** Это НЕ дефект якоря — у документа нет вариант-семей. Идея на будущее (carry): использовать `parent_name_hint`=раздел как группировку/секцию, а не как родителя (сейчас раздел игнорируется). Вне скоупа этого брифа.
6. **Захваченные JSON Gemini** лежат в `scripts/.cache-ai-hierarchy/` (нужны для воспроизводимости `node scripts/test-parser-ai-hierarchy.mjs`; рефреш — флаг `--refresh`). По умолчанию НЕ коммичу (живой выход LLM); скажите, если нужно зафиксировать как фикстуры.
