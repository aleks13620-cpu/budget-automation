# План поэтапного исправления замечаний (<=4 ч на фазу)

> Проект: **Для обочения Промышленности** (production, id=8)  
> Режим исполнения: **строго последовательно**  
> Правило перехода: **следующая фаза только после проверки + коммита текущей**

---

## Правила исполнения (обязательные)

- Каждая фаза ограничена **до 4 человеко-часов** (включая локальные тесты и проверку).
- Внутри фазы: **реализация -> проверка -> фиксация статуса -> git commit**.
- Без выполнения чекпоинтов фазы переход в следующую фазу запрещен.
- После каждой фазы обязательно обновить этот файл: отметить чекбоксы и заполнить поле результата.

### Stop/Go критерий между фазами

- **Go**: все чекбоксы фазы закрыты, проверки пройдены, коммит сделан.
- **Stop**: есть регрессия, незакрытые проверки, или неясный результат по DoD.
- При **Stop** создается подфаза `Xb` (до 4 ч) только на закрытие хвостов текущей фазы.

### Шаблон отметки выполнения (использовать в конце каждой фазы)

- [ ] Фаза реализована в коде
- [ ] Проверка пройдена по чек-листу фазы
- [ ] Регрессия критичных сценариев не выявлена
- [ ] Коммит создан
- [ ] Разрешен переход к следующей фазе

**NOTE:** Если хотя бы один пункт не закрыт, фаза считается незавершенной.

## Фаза 2 (<=4 ч): Точность основного выбора (DN/количество/ранжирование)

### Риск
**Средний-высокий** (изменение scorer может дать регрессии на других проектах).

### Цель
Снизить ошибки, когда система выбирает нерелевантный primary-матч при наличии корректной альтернативы.

### Объем работ
- Усилить scoring в матчинге:
  - бонус за совпадение количества;
  - штраф за mismatch по количеству;
  - приоритет точного DN/диаметра;
  - штраф за конфликт DN.
- Обновить tie-breakers выбора `is_selected`:
  - сначала критичные атрибуты (DN, quantity), затем confidence.
- Гарантировать уникальность выбранного primary-кандидата для каждой spec-позиции.

### Затрагиваемые файлы
- `backend/src/services/matcher.ts`
- `backend/src/routes/matching.ts`

### Проверка фазы
- `Кран шаровой ∅25, 10 шт` -> в primary выбирается DN25 с релевантным количеством.
- `Клапан балансировочный ∅15, 2 шт` -> в primary выбирается корректный клапан DN15 qty2.
- Сценарии вида `8 шт -> 6 шт` перестают попадать в основной выбор при наличии точного количества.

### Триггеры на подфазу 2b
- Падает качество на контрольном проекте вне `id=8`.
- Появляются новые ложные primary-матчи при близких наименованиях.

### Коммит фазы
- Рекомендованный формат:  
  `feat(matching): phase 2 dn and quantity priority in primary ranking`

### Статус фазы 2
- [x] Фаза реализована в коде
- [ ] Проверка пройдена по чек-листу фазы
- [ ] Регрессия критичных сценариев не выявлена
- [ ] Коммит создан
- [ ] Разрешен переход к следующей фазе

**Результат фазы 2:** Усилен scoring в `matcher`: добавлены бонусы/штрафы для quantity и DN (включая штраф за конфликт DN и mismatch quantity). Изменен tie-breaker primary: выбор `is_selected` теперь делается по приоритету DN -> quantity -> confidence, с гарантией одного primary-кандидата на spec-позицию при автосопоставлении. Нужна проверка на целевых кейсах и контрольном проекте перед закрытием фазы.

## Фаза 3 (<=4 ч): Корректный расчет цены (агрегированные количества)

### Риск
**Высокий** (финансы, расхождения между `matching/summary/export`).

### Цель
Убрать финансовые ошибки из-за неверной базы цены при сопоставлении 1 строки счета к нескольким строкам спецификации.

### Объем работ
- Ввести единичную цену с НДС для расчетов:
  - `unit_price_with_vat = line_total_with_vat / invoice_quantity`.
- В summary/export считать сумму по спецификации от единичной цены:
  - `spec_amount = unit_price_with_vat * spec_quantity`.
- Явно маркировать источник цены (raw / derived unit).

### Затрагиваемые файлы
- `backend/src/routes/matching.ts`
- `backend/src/routes/invoices.ts`
- `backend/src/routes/export.ts`

### Проверка фазы
- Кейс 2x8 против строки счета на 16 не дает двойного завышения.
- Для "кожух..." в расчете используется единичная цена с НДС, а не вся строка.
- Итоги `summary` и выгрузка `export` совпадают по логике расчета.

### Триггеры на подфазу 3b
- Любое расхождение сумм между API и export.
- Невоспроизводимость расчета единичной цены по одной и той же строке счета.

### Коммит фазы
- Рекомендованный формат:  
  `fix(pricing): phase 3 unit vat price for aggregated invoice rows`

**Результат фазы 3:** Пересчитана ценовая база для summary/export/matching: при наличии агрегированной строки счета используется `unit_price_with_vat = line_total_with_vat / invoice_quantity`, а сумма по спецификации считается как `unit_price_with_vat * spec_quantity`.
