# План: field-level quality report без изменения парсинга

## Контекст

Сейчас benchmark уже показывает общую картину по OCR: сколько строк найдено, насколько совпали названия, насколько совпали цены и какой общий score. Этого мало для безопасной стабилизации финансового результата. Если документ провалился, нужно быстро понять причину: цена, количество, единица измерения, НДС, сумма строки, итог документа или неверное сопоставление строки с эталоном.

Эта фича — только аналитический слой над существующими benchmark-данными и уже полученными OCR/Gemini-результатами. Она не должна улучшать парсинг, чинить OCR, менять БД, UI или production API.

Связанные источники:
- `scripts/ocr-benchmark/03_score_results.py`
- `scripts/quality-monitor.py`
- `scripts/ocr-benchmark/results/gemini_results.json`
- `scripts/benchmark-ready/train/`
- `scripts/benchmark-ready/holdout/`
- `docs/benchmark-reports/baseline-2026-05-04.md`

Запрещённая зона для этой фичи:
- `backend/src/services/pdfParser.ts`
- `backend/src/services/excelInvoiceParser.ts`
- `backend/src/services/gigachatParser.ts`
- `backend/src/services/invoiceRouter.ts`
- routes, frontend, database schema
- любые новые внешние API-вызовы

## Критерий фича готова

Фича готова, когда есть воспроизводимый отчёт, который:

- считает старые метрики без изменения их смысла: строки, названия, цены, общий score;
- отдельно показывает финансово важные поля: цена, количество, единица, НДС, сумма строки, итог документа;
- для каждого поля различает состояния:
  - `OK` — данные есть и совпали;
  - `FAIL` — данные есть, но не совпали;
  - `MISSING_REF` — поля нет в эталоне;
  - `MISSING_RESULT` — поле есть в эталоне, но отсутствует в результате OCR;
  - `NOT_APPLICABLE` — поле честно неприменимо;
- отдельно показывает известные проблемные документы: `Электротехмонтаж`, `САНТЕХПРОМ`, category C / сканы;
- сортирует проблемы по бизнес-риску: сначала деньги, потом количество/единицы, потом названия;
- по отчёту за 30 секунд понятно, что чинить следующим: цену, количество, единицы, НДС, итог или сопоставление строк;
- обязательно прогоняется на `train` и `holdout` раздельно;
- явно пишет, что parser-код, production API, UI, БД и внешние API не менялись.

Если большая часть финансовых метрик получила `MISSING_REF`, фича не считается полностью готовой как quality gate. В этом случае результатом считается не PASS, а список пробелов эталона, которые нужно закрыть отдельной задачей.

## Риски

- Эталонные JSON могут быть неполными или ошибочными. Неверный эталон заставит считать правильный результат ошибкой.
- Неверное сопоставление строк портит все последующие field-level выводы по цене, количеству, единице и суммам.
- Упаковки, комплекты и агрегированные строки могут давать формальные расхождения при бизнес-корректном результате.
- `MISSING_REF` может скрыть отсутствие критичных финансовых данных в эталоне. Такие случаи должны попадать в `benchmark gaps`.
- Слишком широкий отчёт может стать шумом. Поэтому худшие финансовые риски должны быть сверху.
- Есть риск случайно начать чинить парсер во время реализации отчёта. Это запрещено рамками фичи.
- Есть риск случайно сделать новые платные AI-вызовы. Фича должна работать только на существующих JSON.
- Есть риск подгонки под `train`. Поэтому `holdout` обязателен и показывается отдельно.

## Итог

Эта фича нужна не ради новых процентов в отчёте. Её смысл — быстро ответить на практический вопрос: что именно ломает качество следующего бюджета — цена, количество, единица, НДС, итог, скан или неверное сопоставление строк.

План после правок короче: четыре фазы вместо пяти. Подготовка объединена в один контракт, преждевременный hard gate убран, добавлена проверка на известных проблемных документах, усилены критерии “не менять парсинг” и “не делать внешние API-вызовы”, а старый baseline явно отделён от нового field-level baseline.
