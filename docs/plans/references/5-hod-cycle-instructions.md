# 5-ходовый цикл проверки фазы

Инструкция для оркестратора. Применяется после того как исполнитель отрапортовал «фаза готова».

---

## Принципы

- **Каждый ход — отдельный субагент** с чистым контекстом (без памяти предыдущих ходов). Ход 1 не знает про Ход 2 и не аутосуггестит «всё ок».
- **На каждый ход — свой task/shell-вызов.** В одном ходе может быть несколько подпромптов (1a, 1b, 1c — три подряд).
- **Не верить на слово.** Исполнитель сказал «фаза готова» → гоним цикл.
- **Fix-after-FAIL без перезапуска цикла.** Если в любом ходе найдено «починить» — запускаем fixing-субагент, затем проверяем ТОЛЬКО закрытие найденных пунктов. НЕ гоним весь цикл с нуля.
- **Scope не расширяем.** Если в ходе цикла всплыло «надо бы ещё вот это» → carry-task в новый план, не расширение текущей фазы.
- **После Хода 5** оркестратор сводит итог. Если всё чисто — коммит фазы.

---

## Ход 1. Найди ошибки (три промпта подряд)

**Цель:** свежий взгляд на код/файлы фазы. Найти реальные ошибки, не теоретические риски.

### 1a. Поиск ошибок
Темплейт промпта субагенту:
```
Прочитай свои изменения свежим взглядом. Найди ошибки.

Категории:
- off-by-one (индексы массивов, _parentIndex)
- null / undefined (неинициализированные поля, пропущенные проверки)
- edge cases (пустой массив, одна строка, очень длинные имена, кириллица + латиница)
- race / concurrency (если применимо)
- утечки ресурсов (file handles, db connections, сетевые сессии)
- нарушения инвариантов из architecture/ (parent_item_id, full_name, position_number, confidence range 0..1)

Нет ошибок — назови 3 места, где сломается при росте нагрузки в 100 раз.
```

### 1b. Докажи
Темплейт:
```
Докажи каждый пункт — ошибка или паранойя?
Покажи кодом (цитата строк) или сценарием (входные данные → ожидаемое vs фактическое).
```

### 1c. Исправь
Темплейт:
```
По реальным — лучшее решение и применяй. Не "варианты", а решение. Diff.
```

---

## Ход 2. Что упустили (два промпта)

**Цель:** найти сценарии, которые не покрыты.

### 2a. Поиск упущений
Темплейт:
```
Что упустили?
- сценарии не учли (все комбинации: DN + variant, "То же" + variant, все три типа вместе)
- пограничные случаи (ребёнок без родителя, ребёнок после section header, внутри splitMonsterRow)
- зависимости от других частей кода (matcher, learnConstructionSynonyms, matched_items, parent_item_id в БД, full_name в экспорте)
- что будет через месяц использования (новые шаблоны, накопление данных)
```

### 2b. Докажи
Темплейт:
```
Докажи: реальное упущение в нашем проекте или теоретический риск?
Сошлись на конкретный файл/строку/фикстуру.
```

---

## Ход 3. Это реально проблема? (один промпт)

**Цель:** отсеять теоретические риски, оставить только то что реально мешает продукту.

Темплейт:
```
По каждому пункту — докажи, что РЕАЛЬНО проблема В НАШЕМ проекте.

Учитывай:
- продукт (цели, KPI)
- аудитория (кто пользуется)
- объёмы (текущий поток и целевой)
- приоритет (MVP → продать → масштабировать)
- что уже сделано (история ретроспектив)

"Теоретически может, но не критично" — вычеркни. Оставь только то,
что реально мешает достичь KPI на текущем потоке.
```

**Вердикты:** ИСПРАВИТЬ / CARRY-TASK / ВЫЧЕРКНУТЬ

---

## Ход 4. Свежий взгляд после правок (один промпт)

**Цель:** после всех фиксов проверить что ничего не сломали.

Темплейт:
```
После правок — свежим взглядом.
- что появилось НОВОГО (новые ветки кода, новые файлы, новые таблицы БД, новые env)
- что могли сломать — регрессии:
  * смежные ветки кода (Excel-ветка, другие парсеры)
  * существующие фикстуры (все ли проходят как раньше)
  * build (tsc зелёный?)
  * runner (exit code?)
- scope creep (файлы вне scope фазы?)

Только новое. Если новых рисков нет — явно скажи "регрессий нет, scope чист".
```

---

## Ход 5. Security review (один промпт)

**Цель:** проверить безопасность изменений.

Темплейт:
```
Security review последних изменений.

Проверь:
- ОПАСНАЯ ТРОЙКА (личные данные + чужой контент + отправка наружу)
- SQL/NoSQL injection
- XSS / CSRF / SSRF
- утечки токенов в логи (GIGACHAT_AUTH_KEY, OPENROUTER_API_KEY)
- .env в коде или коммитах
- права на API (новые npm-скрипты не открывают доступ извне)
- rate limiting (если не применимо — явно отметь)
- валидация входов (размеры файлов, корректный JSON)

По каждому: OK / починить — что / не применимо.
```

---

## Укороченный цикл (для документации)

Если фаза — только документ (без кода):
- Ходы 1, 3, 4 — обязательны
- Ход 2 (упущенные сценарии) — опционален (кода нет)
- Ход 5 (security) — опционален (у документа нет поверхности атаки)

---

## Закрытие цикла

После Хода 5 оркестратор:
1. Сводит итог: что исправлено, что carry-task, что вычеркнуто.
2. Если были «починить» → проверяет ТОЛЬКО закрытие найденных пунктов (не перезапускает цикл).
3. Делает коммит фазы (одна фаза = один коммит).
4. Для документов: проверяет язык (русский), portability (нет `c:\...`), двусторонние ссылки.
