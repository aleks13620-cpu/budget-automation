# Worker-бриф — Порча названий при парсинге (битый шрифт) → бьёт по матчингу

> Отдельная задача (оркестратор+worker, по правилам проекта). Реальный сигнал от оператора.
> Создан 2026-06-01. Investigation-first.

## Контракт (зачем)
Названия позиций — ключ матчинга и обучения. Если имя приходит мусором и **по-разному в разных строках**, матчер не узнаёт позицию, а обучение засоряется → падает доля автосопоставления (kill-метрика «ад форматов»). Это НЕ косметика.

## Сигнал (реальный)
Оператор «Инженер Сети Иван»: «слово радиатор по-разному пишется в распознавании счетов». Счёт **ИТЕСА №3227**, проект 11 «Ласточка ОВ», **invoice id=59**, 186 позиций.

## Симптом (проверено read-only на проде 2026-06-01, API `/api/invoices/59`)
Слово «Радиатор» хранится с порчей, варьирующейся по строкам:
```
P?\xa00а0диатор стальной EVRA Compact C 33-600-1000 ...   (Compact C)
P?\xa0Kа0диатор стальной EVRA Ventil Compact CV ...        (CV  → K)
P?\xa0Lа0диатор стальной EVRA Ventil Compact CVL ...       (CVL → L)
```
- Внутри слова: **неразрывный пробел `\xa0` (NBSP) + посторонние символы**; начало «Ра…» искажено.
- **Артикулы (F0Z…/F0W…) извлекаются ЧИСТО** — ломается именно кириллическое НАЗВАНИЕ.
- Признак **битого/встроенного шрифта PDF без нормального ToUnicode** на кириллице (класс «Элита»).

## Корень: слепое пятно роутинга
Счёт сохранён **category A, review=0**, потому что **финансы (qty/price/amount) извлеклись верно**, а эвристика `needsGigaChat = items.length===0 || financialCoreMissing` (`invoices.ts:~521`) роутит в GigaChat при отсутствии ФИНАНСОВ — **но НЕ при порче ИМЁН**. Мусорные имена проходят «успешно» и тихо отравляют матчинг/обучение.

## Investigation-first (read-only, СТОП → вернуть оркестратору)
1. **Точные байты:** забрать сырой JSON/hex названия с прода (`/api/invoices/59`) — отделить реальную порчу от консольной мойибаки. Зафиксировать, какие именно символы лишние.
2. **PDF ИТЕСА:** есть ли ToUnicode у шрифта названий? Битый ли шрифт? (pdffonts/чтение fitz).
3. **Масштаб:** на скольких поставщиках/счетах прода встречается порча имён (read-only) — насколько распространено.
4. **Дизайн фикса** → чекпоинт, ждать «ок».

## Возможный фикс (по результату расследования — НЕ кодить вперёд дизайна)
- **Детектор «мусорного имени»** (NBSP/replacement-char/непечатаемые/контрол-символы внутри слова; доля «битых» имён в счёте) → при срабатывании **роутить в GigaChat** (штатный fallback для нечитаемых шрифтов, как в пресс-релизе) ИЛИ ставить `needs_amount_review`/`needs_unit_review` с причиной. Закрыть слепое пятно: «имена битые» = тоже повод не считать парс успешным.
- **No-hardcode:** детект по ПРИЗНАКУ порчи, НЕ по слову «радиатор» и НЕ по поставщику «ИТЕСА».
- TS+Python синхронно, если затрагивает оба парсера. Forward-only (НЕ reparse). Excel-путь учесть.

## Критерии приёмки
- Битые-имена счета больше не проходят как «успешные» молча: либо чистое имя (через GigaChat), либо явный флаг на review.
- Не ломает счета с нормальными именами (регресс-гейт OLD-vs-NEW на реальном корпусе, как у #4).
- No-hardcode; `cd backend && npm run build` чистый; 5-move гейт PASS; деплой-верификация (SHA+CREATED+health).
