## Модель параллельной работы (оркестратор + worker-чаты)

**Проблема:** несколько чатов на одном репо конфликтуют через общие hot-файлы (`matcher.ts`, `routes/matching.ts`, `MatchTable.tsx`).

**Решение — git worktree на изоляцию (уже используется: `peaceful-booth-3e4dc3`):**

```
Оркестратор-чат (этот)
├── держит: roadmap, метрики, гейт pre-deploy-check, мердж в main
├── НЕ пишет код сам — раздаёт worker-чатам
└── Worker-чаты (каждый в своём git worktree / своей ветке):
    ├── Worker A: парсер цены — backend/src/services/{pdfParser,excelInvoiceParser}.ts  [ИЗОЛИРОВАН ✅]
    ├── Worker B: аналоги UI — MatchTable.tsx + matcher.ts  [⚠️ hot, конфликтует с C]
    └── Worker C: приоритизация — routes/matching.ts summary + MatchingView.tsx  [⚠️ hot]
```

**Правила параллелизации (чтобы не было merge-ада):**
1. **Параллелить можно только задачи с непересекающимися файлами.** Парсер (A) изолирован — безопасно параллелить с чем угодно. B и C обе трогают frontend/matching → НЕ параллелить между собой.
2. **Каждый worker — отдельный git worktree** (`git worktree add ../ba-parser parser-branch`), не отдельный clone. Так общий `.git`, но изолированный checkout.
3. **matcher.ts — священный.** Не трогать без классификатора unmatched (reflection 2026-05-20). Если worker B хочет matcher.ts — сначала классификатор, согласовать с оркестратором.
4. **Гейт у оркестратора:** worker готовит ветку → оркестратор гонит pre-deploy-check 5-move (5 субагентов) → мердж в main. Worker НЕ пушит в main сам.
5. **Один writer на файл в момент времени.** Оркестратор ведёт реестр «какой worker какие файлы держит».

**Практичный старт (рекомендация):** начать с ОДНОГО worker-чата на парсер (изолирован, высокий ROI, H3 подтверждена). Когда освоите ритм worktree+gate — добавлять параллельные. Не начинать 3 чата сразу — сначала отладить процесс на одном.
