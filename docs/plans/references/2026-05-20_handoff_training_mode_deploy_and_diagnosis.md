## Корневой вывод по продукту

Сопоставление спек↔счёт по факту распадается на **три класса**:

| Класс | Что нужно для матча | Доля в текущей задаче (113 unmatched) |
|---|---|---|
| **1. То же написано иначе** | Текст-матчер + правила/синонимы | Малая часть, **уже работает** через task 4 (training mode) |
| **2. Функциональные аналоги** (VMA → GROSS GR4) | Семантическое понимание (LLM) | **~54%** (≈61 кейсов) |
| **3. Чего реально не купили** | Никакая система не поможет | **~46%** (≈52 кейсов, отсутствуют в эталоне) |

LLM (Gemini) уже подключён в матчере как fallback (tier 4). Узкое место — UX вокруг класса 2: LLM-матчи аналогов сейчас выглядят как «требует проверки», но не подсвечены как «возможно аналог другого бренда».

- Не править `matcher.ts` без классификации unmatched по корзинам NO_RULE / CURRENT_BUG / CONTAINMENT_HELPS / ORPHAN_INV. Сегодняшняя ошибка — предлагал containment-фикс **до** анализа данных. Доказательно — 0 кейсов закрыл бы.
- Не наращивать слои в текущем training mode. Он закрывает класс 1, для класса 2 нужен другой UX (см. ниже).
- Не пытаться расширять промпт LLM «вслепую» — он уже работает приемлемо; узкое место — отображение и подтверждение его результата.
