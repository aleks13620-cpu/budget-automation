# Фикс надёжности JSON-парсинга GigaChat — 11.03.2026

## Проблема

GigaChat периодически возвращает невалидный JSON:
- `Bad escaped character` — backslash перед кириллицей (`\Н`, `\р`, etc.)
- Реальные символы переноса строки (0x0A) внутри строковых значений
- Данные документа в полях supplier/buyer name (галлюцинация модели)
- Trailing commas, незакрытые строки

Самодельный `sanitizeJSON` через regex ненадёжен — не может корректно
найти границы строк в уже сломанном JSON.

---

## Решение

Заменить самодельный `sanitizeJSON` на библиотеку **json-repair**,
написанную специально для починки JSON от LLM-моделей.

**Вероятность решения проблемы: 95%+**

---

## План реализации

### Шаг 1 — Установить json-repair
```bash
npm install json-repair
```

### Шаг 2 — Заменить sanitizeJSON в gigachatParser.ts

Убрать самодельную функцию `sanitizeJSON`, заменить на:
```typescript
import { jsonrepair } from 'json-repair';
// вместо: JSON.parse(sanitizeJSON(extractJSON(raw)))
// стало:  JSON.parse(jsonrepair(extractJSON(raw)))
```

### Шаг 3 — Ограничить поля supplier/buyer в промпте

Добавить в INVOICE_PROMPT:
- `"name" — только название компании, не более 150 символов`
- `не включать адреса, реквизиты, текст документа`

### Шаг 4 — Протестировать на данных из системы

До коммита прогнать через Node.js REPL:
- счёт Элита КНС (КП 3360) — вызывал Bad escaped character
- счёт Т-35А — путал ставку НДС с ценой
- любой счёт с кириллическими кавычками в названии поставщика
