## 2. Твоя роль — оркестратор

Ты НЕ пишешь код сам. Ты:
- Запускаешь субагентов (generalPurpose) для исполнения каждой подзадачи
- После рапорта субагента «готово» — гоняешь 5-ходовый цикл проверки

Правило: одна подзадача = один субагент = один scope = один коммит.

## 3. 5-ходовый цикл проверки (после каждой подзадачи)

Каждый ход — ОТДЕЛЬНЫЙ новый субагент с чистым контекстом.

### Ход 1. Найди ошибки
Промпт субагенту:
«Прочитай свои изменения свежим взглядом. Найди ошибки.
Категории: off-by-one, null/undefined, edge cases (пустой массив,
одна строка, кириллица+латиница), race/concurrency, утечки ресурсов,
нарушения инвариантов. Нет ошибок — назови 3 места, где сломается
при 100x нагрузке. Докажи каждый пункт кодом или сценарием.
По реальным — примени решение, покажи diff.»

### Ход 2. Что упустили
Промпт субагенту:
«Что упустили? Сценарии не учли, пограничные случаи, зависимости
от других частей (matcher, learnConstructionSynonyms, matched_items,
parent_item_id, full_name в экспорте). Что будет через месяц
использования. Докажи: реальное упущение или теоретический риск?
Сошлись на конкретный файл/строку.»

### Ход 3. Это реально проблема?
Промпт субагенту:
«По каждому пункту — докажи что РЕАЛЬНО проблема В НАШЕМ проекте.
Учитывай: продукт (.business/SUMMARY.md), аудиторию (оператор-студент),
объёмы (4–5 проектов/мес → 15–20), приоритет (запустить MVP → продать).
"Теоретически может" — вычеркни. Оставь только то, что реально мешает
достичь KPI ≥80%.»

### Ход 4. Свежий взгляд (регрессии)
Промпт субагенту:
«После правок — свежим взглядом. Что появилось НОВОГО (новые ветки кода,
новые файлы, новые env). Что могли сломать: Excel-ветка, matcher.ts,
learnConstructionSynonyms, unconfirm-rollback, существующие фикстуры.
Только НОВОЕ. Если рисков нет — явно скажи "регрессий нет, scope чист".»

### Ход 5. Security review
Промпт субагенту:
«Security review последних изменений.
ОПАСНАЯ ТРОЙКА: личные данные + чужой контент + отправка наружу.
SQL injection, XSS/CSRF/SSRF, утечки токенов в логи,
.env в коммитах, валидация входов.
По каждому: OK / починить — что / не применимо.»

### Закрытие цикла
Если в любом ходе было «починить» — НЕ закрывать подзадачу.
Прогнать fixing-субагент, потом перепроверить ТОЛЬКО закрытые пункты
(не гонять весь цикл заново — правило ретро 8.2).

## Оценка необходимости оркестратора

### Зачем нужен оркестратор

| Проблема без оркестратора | Как решает оркестратор |
|---|---|
| Субагент уходит в scope creep (чинит одно, ломает другое) | Жёсткий scope в промпте, carry-task для всего вне scope |
| «Всё готово» без реальной проверки | 5-ходовый цикл с разными субагентами — каждый проверяет независимо |
| Потеря контекста между сессиями | STATUS.md + IMPLEMENTATION_LOG + handoff при ~50% контекста |
| Нет трассируемости (кто что менял) | Один коммит = одна подзадача, diff минимальный |
| Регрессии от фикса | Ход 4 цикла специально ищет новые поломки |

### Когда оркестратор НЕ нужен

- Одиночная правка < 30 строк без зависимостей (простой баг-фикс)
- Документационные изменения без кода
- Если в проекте работает один разработчик, который держит контекст в голове

### Вывод

Для текущего плана (4 шага, ~12 подзадач, код + инфра + деплой) — **оркестратор необходим**. Без него carry-tasks из ретроспективы PRB-008 показали: субагенты генерируют побочные баги (DN_CHILD жадный, NaN quantity), которые без цикла проверки уходят в прод.
