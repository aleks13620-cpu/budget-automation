## Этап 1 — M1: Matching Rules + Метрики (≤4 ч)

### Задачи

**1.1 Усилить применение matching_rules в матчере**
- Проверить что `matching_rules` реально применяются в Tier2
- Добавить логирование какой tier сработал для каждого матча
- Добавить отрицательные правила (оператор отклонил → не предлагать снова)

**1.2 Импорт эталонной сметы из Excel**
- Новый endpoint `POST /api/projects/:id/import-matches` — принимает Excel
- Формат: колонка «Наименование спецификации» + «Наименование в счёте» + «Поставщик»
- При импорте → создаёт `matching_rules` для всех строк
- UI: кнопка «Импортировать эталонные матчи» в ProjectDetail

**1.3 Дашборд качества матчинга**
- В MatchingView добавить блок статистики:
  - Всего позиций / Авто-сопоставлено / Подтверждено / Не найдено
  - % по каждому tier (rules / similarity / GigaChat)
- В ProjectDetail: прогресс-бар покрытия спецификации

### Коммиты
```
feat: matcher — strengthen Tier2 rules, add negative rules, tier logging
feat: import-matches endpoint — bulk matching_rules from Excel
feat: matching dashboard — coverage stats, tier breakdown
```

## Этап 2 — Feedback + GigaChat валидация (≤4 ч)

### Задачи

**2.1 Сбор обратной связи оператора**
- Новая таблица `operator_feedback (id, type, project_id, spec_item_id, invoice_item_id, comment, created_at)`
- После каждого ручного действия → попап «Система запомнила. Применить к похожим? [Да] [Нет] [Сообщить об ошибке]»
- Кнопка «Сообщить об ошибке» → мини-форма: тип проблемы + комментарий
- Страница `/feedback` — список накопленных сигналов для администратора

**2.2 GigaChat валидация сложных случаев**
- В матчере Tier5: если score < 40% → отправить пару в GigaChat
  - Вопрос: «Это одна позиция? Спецификация: "...". Счёт: "...". Ответь да/нет»
- Применять только к топ-3 кандидатам с низким score
- Результат кэшировать в БД чтобы не повторять запросы

**2.3 Версионирование matching_rules**
- Добавить `created_at`, `source` (manual/import/gigachat), `is_negative` в matching_rules
- История изменений правил — возможность откатить

### Коммиты
```
feat: operator_feedback table + UI popup after manual match
feat: feedback page — admin view of collected signals
feat: matcher Tier5 — GigaChat validation for low-score candidates
feat: matching_rules versioning — source, is_negative, history
```

## Этап 3 — Эмбеддинги: часть 1 (≤4 ч)

> ⚠️ Реализовать только после накопления 3-5 проектов с подтверждёнными матчами

### Задачи

**3.1 Генерация и хранение эмбеддингов**
- Новая таблица `embeddings (id, entity_type, entity_id, vector BLOB, model, created_at)`
- При загрузке спецификации → генерировать эмбеддинги через GigaChat Embeddings API

- Косинусное расстояние — порог 0.82+
- Кэшировать результаты поиска

### Коммиты
```

feat: batch embedding generation for spec and invoice items
```

## Сигналы обратной связи оператора

| Действие | Что запоминает система |
|---------|----------------------|
| Подтвердил матч ✅ | Позитивное правило A→B |
| Отклонил матч ❌ | Негативное правило A≠B |
| Выбрал другую позицию вручную | A→C позитивное + A≠B негативное |
| Исправил название в редакторе | Нормализация названия |
| Применил пересчёт цены | Правило поставщика: цена=сумма÷кол-во |
| Нажал «Сообщить об ошибке» | Запись в operator_feedback |
