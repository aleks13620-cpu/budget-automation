## Этап 1 — Регрессия GigaChat + кнопка редактирования (~3 ч)

### Проблема А: Регрессия — Элита КНС и другие счета парсятся хуже

**Причина:** `invoices.ts` строки 227–229:
```typescript
const needsGigaChat = parseResult.items.length === 0 ||
  (ext === '.pdf' && (quickPdfCategory === 'B' || quickPdfCategory === 'C')) || ...
```
Если классический парсер нашёл позиции, но категория B (пограничная) — GigaChat **перезаписывает** результат. Раньше этой логики не было.

**Исправление в `invoices.ts`:**
- GigaChat вызывается ТОЛЬКО при `items.length === 0` ИЛИ category `C`
- При category B + items > 0 — классический результат сохраняется

### Проблема Б: Нет кнопки редактирования у проверенных счетов

**Причина:** `ProjectDetail.tsx` строка 611 — при `isVerified = true` показывается только "Позиции", все кнопки управления скрыты.

**Исправление в `ProjectDetail.tsx`:**
- Для верифицированных счетов добавить кнопку "Редактировать" → `onInvoicePreview(inv.id)`
- В развёрнутом блоке позиций показывать кнопки действий независимо от статуса

**Файлы:**
- `backend/src/routes/invoices.ts`
- `frontend/src/pages/ProjectDetail.tsx`

**Коммит:** `fix: gigachat override only on C/empty + edit button for verified invoices`

## Этап 2 — НДС: сохранение и отображение (~3.5 ч)

### Проблема: НДС не сохраняется — общая ошибка всех счетов

**Причина:** GigaChat извлекает `vat_amount`, но:
1. В таблице `invoices` нет колонки `vat_amount`
2. В INSERT в `invoices.ts` поле не передаётся
3. В UI (`ProjectDetail.tsx`) поле не отображается

**Исправление:**

1. `backend/src/database/init.ts` — добавить `ALTER TABLE invoices ADD COLUMN vat_amount REAL` (если не существует)
2. `backend/src/routes/invoices.ts` — в INSERT и UPDATE передавать `vatAmount` из GigaChat
3. `frontend/src/pages/ProjectDetail.tsx` — в блоке позиций рядом с итоговой суммой показывать "в т.ч. НДС: X ₽"
4. `frontend/src/pages/ProjectDetail.tsx` — в интерфейсе `Invoice` добавить поле `vat_amount`

**Файлы:**
- `backend/src/database/init.ts`
- `backend/src/routes/invoices.ts`
- `frontend/src/pages/ProjectDetail.tsx`

**Коммит:** `feat: store and display vat_amount from GigaChat parse`

## Этап 3 — Качество: document_type guard + fuzzy supplier (~3 ч)

### Проблема А: GigaChat парсит технические описания как счета

**Причина:** Нет проверки `document_type` из ответа GigaChat. Документ типа "спецификация" или "техническое описание" принимается как счёт.

**Исправление в `gigachatParser.ts` и `invoices.ts`:**
- После парса GigaChat — если `document_type` содержит "спецификация", "техническое", "КП" и нет суммы → не перезаписывать результат классического парсера
- Добавить предупреждение в `parsingCategoryReason`: "Документ не является счётом (тип: ...)"

### Проблема Б: Неправильный поставщик (Фрегат → ОАО АКБ)

**Причина:** GigaChat берёт имя поставщика из текста, но не сверяется с существующими поставщиками в БД.

**Исправление в `invoices.ts`:**
- После получения `supplierName` от GigaChat — fuzzy-match по `suppliers` из БД (уже есть `string-similarity`)
- Если сходство > 0.75 — использовать существующее имя поставщика
- Если < 0.75 — создать нового поставщика с именем от GigaChat

**Файлы:**
- `backend/src/services/gigachatParser.ts`
- `backend/src/routes/invoices.ts`

**Коммит:** `fix: document_type guard + fuzzy supplier match after GigaChat`
