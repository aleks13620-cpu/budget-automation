## Этап 1 — json-repair: устойчивый JSON-парсинг (~2 ч)

### Проблема
GigaChat возвращает невалидный JSON: `Bad escaped character`, реальные
переносы строк в строках, trailing commas. Самодельный `sanitizeJSON`
ненадёжен — не может корректно найти границы строк в сломанном JSON.

### Решение
Установить библиотеку **json-repair** (создана специально для LLM-вывода).
Вероятность решения JSON-ошибок: **95%+**

### Задачи
1. `npm install json-repair` в backend
2. Убрать `sanitizeJSON` из `gigachatParser.ts`
3. Заменить `JSON.parse(sanitizeJSON(extractJSON(raw)))` на
   `JSON.parse(jsonrepair(extractJSON(raw)))` — во всех 3 местах
4. Добавить в INVOICE_PROMPT ограничение:
   `supplier.name и buyer.name — только название компании, не более 150 символов`
5. Тест: "Переразобрать (GigaChat)" на счёте Элита КНС (КП 3360)

**Файлы:** `backend/package.json`, `gigachatParser.ts`

## Этап 2 — Few-shot примеры в промпте (~3 ч)

### Проблема
GigaChat учится на примерах лучше чем на правилах. Текущий промпт содержит
только инструкции — нет ни одного показательного примера.

### Решение
Добавить в `INVOICE_PROMPT` 1 полный пример правильного парсинга реального
счёта из системы: счёт с 3–5 позициями, явным НДС, итоговой суммой.
Пример берётся из счёта, который уже верифицирован пользователем.

### Задачи
1. Выбрать из БД верифицированный счёт с хорошим результатом
   (например, счёт ОНТЕХ или ПСТМ Престиж 154 — Алексей отметил как верный)
2. Сформировать пример в формате: `ПРИМЕР ПРАВИЛЬНОГО РЕЗУЛЬТАТА:` + JSON
3. Добавить в конец `INVOICE_PROMPT` перед `ОТВЕТ — ТОЛЬКО JSON:`
4. Тест: Переразобрать 3–4 счёта, сравнить количество позиций и суммы

**Файлы:** `gigachatParser.ts` (INVOICE_PROMPT)

**Прирост точности: +3–5%**

## Этап 3 — Флаг "под сомнением" в UI (~3 ч)

### Проблема
Когда GigaChat возвращает неверные данные (сумма позиций ≠ итогу документа),
система сохраняет их без предупреждения. Пользователь видит ошибку только
при ручной проверке.

### Решение
После парсинга GigaChat — сравнить `Σ(price × quantity)` с `total_with_vat`.
Расхождение > 15% → сохранить флаг `needs_amount_review = 1` в БД и показать
предупреждение в UI.

### Задачи

**Backend:**
1. `init.ts` — добавить колонку `needs_amount_review INTEGER DEFAULT 0` в `invoices`
2. `invoices.ts` — после GigaChat парса вычислить расхождение, записать флаг
3. `invoices.ts` — включить `needs_amount_review` в SELECT списка счетов

**Frontend:**
4. `ProjectDetail.tsx` — добавить поле в интерфейс `Invoice`
5. `ProjectDetail.tsx` — в строке счёта показывать предупреждение
   `⚠ Сумма под сомнением` рядом со статусом (жёлтый, не блокирует)
6. В развёрнутом блоке позиций — показывать расчётную и фактическую сумму

**Файлы:** `init.ts`, `invoices.ts`, `ProjectDetail.tsx`

**Эффект:** Пользователь видит проблему сразу, не после ручного просмотра

## Этап 4 — Supplier-specific контекст (~4 ч)

### Проблема
GigaChat получает одинаковый промпт для всех счетов, не зная специфику
конкретного поставщика: как выглядит их таблица, какие колонки, типичная
структура.

### Решение
Перед отправкой в GigaChat — проверить есть ли у поставщика сохранённый
parser config. Если есть — добавить в промпт блок с описанием структуры его
счетов. Источник данных уже есть в таблице `supplier_parser_configs`.

### Задачи

**Backend:**
1. `gigachatParser.ts` — добавить необязательный параметр `supplierContext`
   в `parsePdfWithGigaChat` и `parseExcelWithGigaChat`
2. `gigachatParser.ts` — если `supplierContext` передан, добавить в промпт блок:
   ```
   КОНТЕКСТ ПОСТАВЩИКА: счёт от "{name}".
   Структура их таблицы: колонки {список колонок из savedMapping}.
   Ориентируйся на эту структуру при извлечении данных.
   ```
3. `invoices.ts` — при наличии `supplierId` и `savedMapping` — формировать
   `supplierContext` и передавать в GigaChat-парсер
4. `invoices.ts` (reparse-gigachat) — то же самое

**Тест:**
5. Загрузить счёт поставщика с сохранённым конфигом
6. Убедиться что лог показывает "Supplier context added"
7. Сравнить качество с предыдущим парсингом

**Файлы:** `gigachatParser.ts`, `invoices.ts`

**Прирост точности: +3–5% для поставщиков с сохранённым конфигом**
