# MVP: Функциональные требования

## Что делает система (пошагово)

### Шаг 1: Создание проекта
- Оператор создаёт проект (название, например "ЖД Межевая")
- Загружает спецификацию Excel (из PDF проекта)
- Система читает позиции: название, артикул, количество, ед. изм.

### Шаг 2: Загрузка счетов
- Оператор загружает счета поставщиков (Excel или PDF)
- Система извлекает позиции с ценами
- Автоматически начинает поиск совпадений со спецификацией

### Шаг 3: Сопоставление (ключевой модуль)
- Зелёный (95%+): точное совпадение — подтверждать одним кликом
- Жёлтый (70–95%): вероятное — оператор проверяет
- Красный: не найдено — вручную или пропустить
- **Обучение:** подтверждённые совпадения запоминаются как правила-синонимы

### Шаг 4: Группировка по разделам
- Ручное назначение или автопредложение по ключевым словам
- Разделы: Отопление, Вентиляция, ВК, Тепломеханика/ИТП, Автоматизация, Кондиционирование, Электрика, Слаботочка

### Шаг 5: Отчёт
- Итоги по разделам + общий итог
- Список позиций без цен
- Детализация по разделам
- Экспорт в Excel

## Ограничения MVP
- PDF: только простые таблицы (сложные/сканы — ручная корректировка)
- Авторасчёт запросов поставщикам — НЕ реализован
- Получение счетов из почты — НЕ реализовано
- Роли: базовая авторизация логин/пароль (без разделения прав)
- Только веб (без мобильной адаптации)
- Нет интеграции с AMO CRM и 1С

## Технические характеристики
- До 10 пользователей одновременно
- 1 проект (50–100 позиций): обработка 2–5 минут
- Доступность: 99%
- Резервное копирование: ежедневно

## Текущая техническая проблема (активная)
GigaChat обрезает JSON при парсинге счётов > ~37 позиций.
Причина: кириллица токенизируется ~2 символа/токен (vs ~4 для латиницы).
Большой счёт (~37 позиций) → ~5000 output-токенов > лимит maxTokens: 4096.
Файл: `gigachatParser.ts`.

### План исправления (4 шага, по порядку):
1. `cleanInvoiceText()` — убрать банковские реквизиты/подписи (~45% сокращение input)
2. Компактный array-based JSON вместо verbose объектов (~50% сокращение output)
3. `mapItems()` — backward-compatible поддержка обоих форматов
4. Поднять `maxTokens: 8192` как страховочный буфер
