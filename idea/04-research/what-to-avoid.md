# Что избегать — где болит у конкурентов и почему провалились

_Конкретные ошибки, задокументированные в рынке. Применяем к Budget Automation._

---

## Причина смерти №1: Wrong ICP — Enterprise вместо SMB

**Кто попался**: Alice Technologies ($30M, закрылась 2023), Constrafor (реструктуризация после $106M), несколько "smart BOM" стартапов 2019–2022.

**Как это выглядело**: Стартап видит большие компании → думает "там деньги" → идёт к Tier 1 GC ($500M+ revenue) → цикл продаж 12–18 месяцев → IT-комитет, юридический отдел, security review → к этому моменту деньги кончились.

**Почему особенно опасно для нас**: В России крупные строительные компании → 1С → их IT-отдел скажет "интегрируем только через 1С API". Там не наш рынок.

**Что делать вместо**: Беречь как зеницу ока **средний строительный подрядчик/субподрядчик** (20–200 человек, 1 сметчик на компанию). У них нет IT-отдела, нет 1С, есть Excel и pdf-счета в почте. Они платят карточкой онлайн. Цикл продажи — один demo call.

## Причина смерти №2: Стоимость нормализации данных выше ожидаемой

**Кто попался**: Большинство "specification matching" стартапов 2019–2022 в solar/construction.

**Как это выглядело**: Демо работает на 5 клиентских файлах. Первые 10 реальных клиентов приносят 50 форматов Excel, которые не похожи на демо. Команда уходит в бесконечный "ещё один шаблон". Продукт не масштабируется.

**Почему применимо к нам**: Budget Automation уже встретил это (отсюда auto-detection заголовка, GigaChat-fallback, per-supplier configs). **Не останавливаться** — каждый новый клиент добавляет новые edge cases. Нужна метрика % успешных auto-parse без ручного вмешательства.

**Что делать**: 
1. Отслеживать % документов, где потребовалась ручная настройка column mapping
2. Цель: 85%+ auto-parse для повторных поставщиков, 60%+ для новых
3. При падении ниже — инвестировать в парсер, не в новые фичи

## Причина смерти №3: Нет сетевого эффекта — данные клиентов изолированы

**Кто попался**: Большинство construction procurement SaaS.

**Как это выглядело**: Каждый клиент обучает "свою" систему. Компания А подтвердила 500 пар "ВВГнг 3х2.5" → её правила не помогают компании Б. Нет compound value, нет retention flywheel.

**Что делать вместо**:
1. Анонимизированный общий словарь синонимов (поставщик × номенклатура — без бизнес-данных)
2. Opt-in: "поделиться правилом с другими пользователями системы"
3. Seed из публичных данных (ФСНБ, прайс-листы открытых поставщиков)

**Дополнительный эффект**: Новый клиент сразу получает 70% совпадений без обучения → AHA moment на первом же проекте.

## Причина смерти №4: Обещали 100% автоматизацию

**Кто попался**: Любой procurement AI стартап, который презентовал "hands-free matching".

**Как это выглядело**: Sales pitch "система сама всё сопоставит" → клиент видит 20% непокрытых позиций → разочарование ("обещали 100%, а тут ручная работа").

**Реальность**: 80% автоматизации — это хорошо. Но если это не было ожидаемым, клиент недоволен даже при экономии 4 часов из 5.

**Что делать**:
1. В onboarding: "система автоматически закроет ~80% позиций, остальные вы подтверждаете за 15 минут"
2. Показывать время, сэкономленное системой (counter "без системы это заняло бы X часов")
3. Метрика успеха: "покрытие 80%+ за <30 минут review" — не "100% автоматизация"

## Причина смерти №5: Слишком рано в API/интеграции, слишком поздно в retention

**Кто попался**: Archdesk, Buildxact (до определённой степени).

**Как это выглядело**: Команда строит "интеграция с Procore", "API для 1С", "webhook для ERP" — до того как retention базовой функции доказан. Интеграции привлекают enterprise, enterprise → пункт 1.

**Что делать**: Не строить API и интеграции до тех пор, пока:
- 10+ клиентов не используют систему >3 месяцев
- Retention >80% после первого проекта
- Есть чёткий ICP (не "все строительные компании")

## Причина смерти №6: Field adoption — ПО не доходит до конечного пользователя

**Кто попался**: Buildots (site monitoring), Alice Technologies, любой construction IoT.

**Как это выглядело**: C-suite купил, IT задеплоил, прорабы/сметчики открыли один раз и вернулись к Excel.

**Почему применимо**: Наш target user (сметчик) 30+ лет работает в Excel + ГРАНД-Смета. Переключение — это pain, не gain, если UX сложный.

**Что делать**:
1. Первый сеанс работы: загрузил файл, увидел результат — не более 5 минут до AHA moment
2. Не требовать обучения системы перед первым matching (использовать базовый fuzzy из коробки)
3. Сохранять возможность "скачать Excel как всегда" — параллельный workflow, не замена
4. Не требовать регистрации для первого демо (если возможно)

## Конкретные антипаттерны для Budget Automation

| Риск | Признак | Защита |
|---|---|---|
| Enterprise creep | Первый крупный клиент просит "интеграцию с 1С" | Отказать или взять отдельный контракт, не в roadmap |
| Format hell | >20% документов требуют ручного column mapping | Инвестировать в auto-detection, не добавлять фичи |
| Automation overcommit | В pitch говорим "полная автоматизация" | Всегда говорить "80% + быстрый review" |
| Data isolation | Правила каждого клиента в его изолированной БД | Строить shared anonymous synonym store с opt-in |
| Feature bloat | Добавляем email-уведомления, API, roles до product-market fit | Не трогать до 10 активных клиентов |
| GigaChat dependency | GigaChat недоступен → система падает | Graceful degradation: LLM fallback опционален, классический парсер работает всегда |
