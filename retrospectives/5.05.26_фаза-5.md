## Что пошло хорошо

- Шаг 5а получился диагностическим и read-only: endpoint только показывает конфликты, не меняет matching_rules.
- Шаг 5б не менял алгоритм матчинга, а добавил трассируемость: matcher возвращает matchingRuleId, route показывает понятный match_reason, UI отображает причину рядом с confidence.
- Security review помог явно зафиксировать архитектурную границу: проект single-tenant, matching_rules — глобальная база знаний компании, поэтому показ сработавшего правила оператору является фичей.
- TypeScript build и plans:check:strict проходят после мержа 5а/5б.

## Что пошло не так / неожиданности

- Агент security review смотрел на систему как на multi-tenant и поднял "cross-project leakage"; оркестратор перепроверил факты и отклонил finding как неприменимый к текущей модели доступа.
- Наблюдение про optional API_SECRET подтвердилось, но оказалось pre-existing и не относится к 5б; его нужно держать как deploy backlog.
- После Python tests появился scripts/tests/__pycache__/, который старый .gitignore не покрывал, потому что игнорировал только scripts/__pycache__/.
- STATUS.md лежит вне репозитория (`C:\Users\home\vscode101\docs\plans\active\STATUS.md`), поэтому его обновление требует отдельного разрешения на запись за пределами workspace.

## Технический долг, выявленный в фазе

- Production deploy должен явно задавать API_SECRET или закрывать API на уровне reverse proxy; docker-compose.yml сейчас его не прокидывает.
- matching_rules остаётся глобальной базой знаний без project_id — это корректно для single-tenant, но станет риском, если появится разграничение клиентов/проектов.
- Conflict endpoint пока только диагностический; процесс разрешения конфликтов правил ещё не автоматизирован.
- Общий ignore для Python cache (`**/__pycache__/`) нужно закоммитить вместе с housekeeping перед переходом дальше.

## Правила, выведенные из фазы

- Security review агента проверять по фактической модели проекта, а не принимать multi-tenant выводы автоматически.
- Для диагностических endpoint начинать с read-only формата и понятного JSON, не добавлять мутации до появления процесса разрешения.
- При добавлении "почему совпало" сохранять id причины в данных (`matching_rule_id`), а не пытаться реконструировать её задним числом.
- Перед закрытием фазы проверять не только код, но и repo hygiene: generated artifacts и cache-директории не должны висеть в git status.
