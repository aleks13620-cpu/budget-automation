## Что пошло хорошо

- Быстро отделили инфраструктурную проблему (доступность сервера) от функциональной (качество PDF-парсинга).
- Найден и исправлен корневой разрыв между Excel и PDF ветками по parent/child логике.
- Добавлен timeout без изменения API-контрактов и без влияния на успешные быстрые запросы.
- Деплой проверен не только по `git pull`, но и по фактическому содержимому `backend/dist` в контейнере.

## Что пошло не так / неожиданности

- Первичный симптом выглядел как "слабый сервер", но CPU/health показывали, что это не bottleneck.
- На сервере `git pull` сначала показывал `Already up to date`, что создало ложное ощущение, что фикс уже применен.
- Прежние правила/исправления по спецификации воспринимались как "игнор", потому что PDF-пайплайн терял parent/child контекст до этапа обогащения.

## Технический долг, выявленный в фазе

- Нужен регрессионный набор PDF-кейсов для parent/child (DN, "То же", параметрические строки) с ожидаемым `full_name`.
- Нужен авто-тест на parity между Excel и PDF ветками для логики parent/child.
- Нужны метрики качества PDF-спеки на реальных файлах (извлеченные позиции, доля дочерних строк с валидным контекстом, доля section-header false positives).

## Правила, выведенные из фазы

- Для LLM Files API обязательно ставить timeout на каждый сетевой запрос, даже если есть retry.
- При добавлении эвристик в одну ветку парсинга (Excel/PDF) проверять parity второй ветки в той же фазе.
- Проверку деплоя фиксировать двойным способом: `git HEAD` + runtime-проверка `dist` внутри контейнера.
- Для багов качества парсинга проверять не только количество строк, но и структуру parent/child (`_parentIndex`, `full_name`).
