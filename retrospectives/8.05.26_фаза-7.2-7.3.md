## Что пошло хорошо

- Исполнитель (Cursor) выдержал границы: backend/, frontend/, database/ не тронуты.
- Скрипт явно пишет в заголовке отчёта что внешних API-вызовов нет — контракт виден без чтения кода.
- `KNOWN_PROBLEM_DOCUMENTS` вынесен в константу — будущие правки не требуют лазить по логике.
- Output path guard (`ensure_output_within_results_dir`) закрывает случайную перезапись файлов вне `results/`.
- Compact mode + `--full-row-details` — отчёт не превращается в простыню при большом числе документов.
- Тест на category C проверяет что имена сканов не обрезаются (`...` не появляется) — поймает регрессию при переименовании файлов.

## Что пошло не так / неожиданности

- `STATUS.md` лежал в `docs/plans/active/` и считался вторым активным планом — plan-guard падал с WIP=2. Проблема была известна из ретроспективы 7.1, но не устранена до этой фазы.
- `git mv` + commit для переноса STATUS.md пользователь пробовал вручную из терминала, но команда не выполнилась (PowerShell / путь) — оркестратор сделал перенос напрямую.
- PowerShell не поддерживает `&&` — это проявилось при попытке запустить `cd backend && npm run plans:check:strict` как одну команду.
- Перед push накопилось 20+ коммитов опережения `origin/main` — технический долг из предыдущих фаз.
- Агент создал корректировки (hardening) поверх уже смерженного коммита — потребовался отдельный fixup-коммит вне ветки.

## Технический долг, выявленный в фазе

- Старые хвосты в рабочей копии до сих пор не закрыты: `.business/INDEX.md`, `.business/SUMMARY.md`, `docs/plans/README.md`, `AI_WORKSPACE/`, `retrospectives/6.05.26_фаза-6.md` — нужна отдельная уборочная задача.
- `line_amount` в эталоне отсутствует (`MISSING_REF`) — пока нельзя считать как финансовую метрику. Нужно дополнить benchmark.
- `document_total`, `vat_rate`, `vat_amount` помечены `NOT_APPLICABLE` — эталон не содержит этих полей.
- row-matching для Электротехмонтаж даёт 3 FAIL — до исправления field-level выводы по цене/количеству для этого документа ненадёжны.
- Фаза 4 плана (новый field-level baseline `train`+`holdout` раздельно) ещё не выполнена.

## Правила, выведенные из фазы

- PowerShell: все команды с `&&` давать двумя отдельными строками — `cd backend`, затем `npm run ...`.
- Если агент делает правки поверх уже смерженной ветки — коммитить fixup прямо в `main`, не создавать новую ветку ради одного исправления.
- `STATUS.md` не должен лежать в `docs/plans/active/` — plan-guard считает все `.md` в этой папке планами.
- Hardening (константы, guards, тесты) делать в том же шаге, что и основную реализацию, а не откладывать.
- Перед сменой инструмента исполнителя (Codex → Cursor) явно обновить правила в памяти оркестратора.
