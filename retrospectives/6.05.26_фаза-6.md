## Что пошло хорошо

- Инструменты 6а/6б закрыли практический цикл обучения: теперь есть отчёт по ошибкам операторов и экспорт новых кандидатов benchmark-файлов.
- parser_overrides сначала сделали как безопасную инфраструктуру, затем отдельным шагом довели до реального исполнения.
- Оркестраторская валидация поймала важные production-риски до merge: cache без supplier context, расхождение upload/reparse, отсутствие row-level проверки и потерю VAT-семантики.
- Фиксы сохранили правило: без overrides обычные поставщики не должны получать лишний GigaChat-вызов.
- После всех merge backend TypeScript build проходит.

## Что пошло не так / неожиданности

- Агенты дважды оставляли незакоммиченные изменения после заявленного завершения: в 6б и 6г пришлось отдельно валидировать и коммитить fixup.
- Первый вариант 6г реализовал happy path, но не закрыл критичные edge cases: повторный reparse, cache с изменившимся context, согласованность `price/quantity/amount`, VAT.
- Агент 6в сделал больше фронта: добавил в план будущий шаг 6г. Это оказалось полезно, но потребовало отдельного решения оркестратора принять docs-follow-up.
- На Windows `npx` через PowerShell блокировался ExecutionPolicy; проверку TypeScript нужно запускать через `npx.cmd`.
- `bash` не был в PATH; проверки shell-скрипта 6б пришлось выполнять через `C:\Program Files\Git\bin\bash.exe`.
- UTF-8 документы при обычном `Get-Content` читались mojibake; для планов/ретро нужно явно использовать `-Encoding UTF8`.

- Для parser_overrides стоит добавить автоматические тесты на merge строк, context-aware cache purpose и VAT-normalization.
- Conflict endpoint из фазы 5 остаётся диагностическим; процесс разрешения конфликтов правил ещё не автоматизирован.

## Правила, выведенные из фазы

- Не принимать PASS от агента, если `git status` грязный, даже если commit уже есть.
- Для LLM/OCR cache ключ обязан учитывать всё, что влияет на prompt/result: supplier context, saved mapping, overrides.
- Upload и reparse должны использовать одну и ту же бизнес-логику парсинга; иначе оператор получает разные результаты для одного файла.
- При слиянии цен из другого источника обязательно проверять строковый инвариант `price * quantity ~= amount`.
- VAT-семантика должна быть явной в metadata; нельзя полагаться на имя поля `price`, если downstream может добавлять НДС.
- На Windows для проверок использовать `npx.cmd`, а shell-скрипты гонять через Git Bash, если `bash` не в PATH.
