## Что пошло хорошо

- Фаза 1 осталась в правильной границе: работали только с существующими benchmark/OCR JSON и документами.
- Контракт явно разделил `OK`, `FAIL`, `MISSING_REF`, `MISSING_RESULT`, `NOT_APPLICABLE`.
- Зафиксированы правила для строк, названий, цен, количества, единиц, НДС, суммы строки и итога документа до реализации отчёта.
- Финансовые поля поставлены выше name score, чтобы будущий отчёт не прятал денежный риск за хорошим совпадением названий.
- Известные проблемные документы вынесены в обязательную зону внимания: `Электротехмонтаж`, `САНТЕХПРОМ`, category C / сканы.
- После ревью агент сделал отдельный fixup-коммит, а не смешал исправления с Фазой 2.

## Что пошло не так / неожиданности

- Первый коммит Фазы 1 оказался не финальным: после `25e5208` остался незакоммиченный diff в contract-файле.
- В первичной версии контракта были рискованные допущения: `total_sum` выглядел как verified document total, `position_count` мог использоваться как row-count truth, а `price_with_vat` сравнивался с Gemini `price` без явной VAT-семантики.
- Граница по category C потребовала уточнения: в Фазе 1 разрешены только `train/`, `holdout/`, `gemini_results.json`; `category_c_list.json` не должен становиться скрытым входом без отдельного расширения boundary.
- В рабочей копии остались старые нерелевантные изменения, поэтому `git status` не clean даже после PASS по Фазе 1.
- `main` локально впереди `origin/main` уже на 20 коммитов; это не мешает локальной фазе, но ухудшает передачу состояния.

## Технический долг, выявленный в фазе

- `plan-guard` всё ещё считает `STATUS.md` вторым active-plan и падает по WIP=1. Это отдельная задача процесса, не часть Фазы 7.1.
- Нужно решить судьбу старых хвостов: `.business/INDEX.md`, `.business/SUMMARY.md`, `docs/plans/README.md`, `AI_WORKSPACE/`, `retrospectives/6.05.26_фаза-6.md`.
- Для будущего baseline нужно не объявлять hard gate, пока финансовые поля с `MISSING_REF` не закрыты расширением эталона.
- `total_sum` в benchmark пока считается metadata, а не verified document total. Нужна отдельная проверка семантики перед использованием как метрики.
- `position_count` может расходиться с `items.length` (`Электротехмонтаж`: 33 против 34), поэтому будущий отчёт обязан показывать benchmark integrity warning.
- Перед командной передачей или деплоем нужно решить вопрос с `main...origin/main [ahead 20]`.

## Правила, выведенные из фазы

- Если после коммита агент продолжает исправлять deliverable, результат фазы не PASS до отдельного commit/fixup.
- Контракт метрик должен запрещать “удобные” сравнения, если нет честной базы: лучше `NOT_APPLICABLE` или `MISSING_REF`, чем ложный процент.
- `total_sum` и похожие audit metadata нельзя автоматически превращать в эталонную финансовую метрику.
- Старые aggregate scores должны остаться legacy-сигналом; field-level financial gate должен жить отдельно.
- Известные проблемные документы должны быть обязательными контрольными примерами, а не случайными строками в общей таблице.
- Перед Фазой 2 исполнитель должен читать contract-файл как источник правил реализации, а не заново придумывать пороги.
